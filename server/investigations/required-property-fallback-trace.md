# Required-Property Fallback Runtime Trace — Investigation Report

## 1. Executive Summary

**Single most likely root cause**: The `isGenericTypeDefault` function in `required-property-enforcer.ts` (line 347) does not recognize empty string `""` as a generic default for `System.Object` types — it only matches `"Nothing"` and `"null"`. The catalog declares `LogMessage.Message` with `clrType=System.Object` (not `System.String`), so when the deterministic generator emits `Message=""`, the enforcer's `checkPropertyValue` never enters the `blockOnGenericDefault` branch (line 1228) and the context-derived fallback registered in `CONTEXT_DERIVED_FALLBACK_REGISTRY` is never reached. This type-mismatch gap at line 361 is the earliest proven loss point in the pipeline. Fixing it would unblock LogMessage.Message (the most commonly failing property) and expose a secondary writeback bug in `injectResolvedPropertyBindings` (line 1755) that also prevents InvokeWorkflowFile.WorkflowFileName from persisting its correctly resolved fallback. The remaining three target properties (ReadTextFile.File, SendSmtpMailMessage.Body, QueryEntity.EntityType) lack registered fallback infrastructure entirely.

## 3. Enforcer Call-Path Analysis

### 3.1 Full vs. Conservative Enforcement

All five activities flow through `enforceForFile` (line 810), not `conservativeEnforceForFile` (line 894). Runtime evidence confirms: enforcer logs show `CONTEXT-DERIVED-FALLBACK`, `INVALID SUBSTITUTION BLOCKED`, and `FALLBACK-POLICY BLOCKED` messages, which are only emitted from the `enforceForFile` → `checkPropertyValue` path (lines 1258, 1316, 1077), not from `conservativeEnforceForFile` (which only detects sentinel values at line 923).

### 3.2 Activity Tag Regex (line 823)

```typescript

```

Verified against emitted XAML: matches `ui:LogMessage`, `ui:InvokeWorkflowFile`, `ui:ReadTextFile`, `umail:SendSmtpMailMessage`, `ds:QueryEntity`. Runtime enforcer output confirms all five are processed.

### 3.3 Namespace-Prefix Stripping (line 832)

```typescript

```

Produces `LogMessage`, `InvokeWorkflowFile`, `ReadTextFile`, `SendSmtpMailMessage`, `QueryEntity` — matching `FALLBACK_ELIGIBLE_POLICIES` and `CONTEXT_DERIVED_FALLBACK_REGISTRY` keys exactly. Runtime logs confirm stripped names in enforcer output.

### 3.4 Child-Element Detection (line 854)

```typescript

Uses `activityTag` (full prefixed tag). ### 3.5 Critical Type Mismatch (LogMessage.Message)

Catalog: `LogMessage.Message clrType=System.Object`. `isGenericTypeDefault("", "System.Object")` (line 361): Object branch checks `"" === "Nothing"` (false) and `"" === "null"` (false) → returns false. `isGenericRaw` is false at line 1225, so `blockOnGenericDefault` at line 1228 is never entered. ### 3.6 Missing Property Detection (ReadTextFile.File)

The enforcer iterates catalog-required properties for `ReadTextFile`. When the `File` property is not found as an attribute (not in `attrs` string) and not found as a child element (`bodySection.includes("ui:ReadTextFile.File")` is false), the enforcer enters `handleMissingRequiredProperty` (L1097). ## 5. Stub/Removal Interaction Analysis

### 5.1 Stub Injection (lines 3947-3958)

`generateStubWorkflow` at line 3947 replaces entire workflow when generators fail. Stubbed workflows contain only a minimal Sequence with no real activities — the enforcer never sees the target activities. However, runtime evidence shows generators succeed and emit valid XAML with empty values, so stubbing is not the failure mode for the five target properties.

### 5.2 Reachability Pruning (lines 5206-5344)

`buildReachabilityGraph` (line 1475) determines reachability from Main.xaml. `removeUnreachableFiles` (line 5318) removes truly orphaned files from `deferredWrites` and `xamlEntries`. Pruned workflows are absent from `postGateXamlEntries` when enforcement runs at line 8372. This does not cause the target property failures — the properties fail in workflows that ARE present in `postGateXamlEntries`.

### 5.3 Late Stub Emission (lines 5955-5988, 7115-7136)

Late stubs for missing manifest entries (line 5955) or corrupted files (line 7115) bypass the enforcer. These inject stubs via `syncXamlWrite` after or alongside enforcement. No interaction with the five target properties' fallback path.

**Conclusion**: No stub injection, reachability pruning, or late stub emission path causes the observed failures. The five target properties fail because their empty/absent values survive through the enforcer without modification — either because the fallback path is unreachable (LogMessage.Message type gap), the writeback is skipped (InvokeWorkflowFile.WorkflowFileName injection bug), or no fallback/emission exists (ReadTextFile.File, SendSmtpMailMessage.Body, QueryEntity.EntityType).

## 6. Minimum-Valid-Emission Gap Analysis

### 6.1 LogMessage.Message
- **Current emission authority**: Deterministic generator `gen_log_message` (deterministic-generators.ts:206) emits `Message=""` when no message value is provided (line 209: `args.message || args.Message || ""`).
- **Enforcer authority**: Full infrastructure exists — `FALLBACK_ELIGIBLE_POLICIES` (line 198) with `blockOnGenericDefault: true`, `contextDerivedFallbackAllowed: true`; `CONTEXT_DERIVED_FALLBACK_REGISTRY` (line 160) with `deriveExpression` that produces `"[Auto-generated] Log: ${label}"`.
- **Gap**: `isGenericTypeDefault` (line 361) does not treat `""` as generic default for `System.Object`. The fallback infrastructure is present but unreachable.
- **Where authority should live**: The enforcer has the correct authority already. The `isGenericTypeDefault` function must recognize empty string for `System.Object` types to unblock the existing fallback path.

### 6.2 InvokeWorkflowFile.WorkflowFileName
- **Current emission authority**: Deterministic generator `gen_invoke_workflow_file` (deterministic-generators.ts:244) emits `WorkflowFileName=""` when no filename is provided (line 246: `args.workflowFileName || args.WorkflowFileName || ""`).
- **Enforcer authority**: Full infrastructure exists and correctly resolves the fallback. Binding created with `source=context-derived resolved="AutoGenerated_Invoke_Invoke_Process.xaml"`.
- **Gap**: `injectResolvedPropertyBindings` (line 1755) skips writeback because the attribute already exists with empty value.
- **Where authority should live**: The enforcer has the correct authority. The injection function must handle replacement of existing empty-value attributes, not just insertion of missing attributes.

### 6.3 ReadTextFile.File
- **Current emission authority**: Deterministic generator `gen_read_text_file` (deterministic-generators.ts:356) emits `FileName` (type `System.String`) but does NOT emit `File` (type `UiPath.Platform.ResourceHandling.IResource`). The activity registry in `uipath-activity-registry.ts` (line 146) lists `["FileName", "Content", "Encoding"]` — `File` is absent.
- **Enforcer authority**: None — no `FALLBACK_ELIGIBLE_POLICIES` entry, no `CONTEXT_DERIVED_FALLBACK_REGISTRY` entry for `ReadTextFile.File`. The enforcer correctly detects the missing required property but has no recovery mechanism.
- **Gap**: Generator/catalog mismatch — the catalog requires `File` (an `IResource` typed property representing a storage-backed resource reference), but the generator only knows about `FileName` (a plain string path). Neither the generator nor the enforcer has authority to produce an `IResource`-typed value.
- **Where authority should live**: The deterministic generator should be updated to emit `File` when it can be derived from `FileName` (wrapping the path in an `IResource` expression), or a new `CONTEXT_DERIVED_FALLBACK_REGISTRY` entry should provide an `IResource`-compatible value. Alternatively, the activity registry should be updated to include `File` so the tree-assembler emits it. The `IResource` type makes auto-generation non-trivial — this may require a dedicated resource-binding helper.

### 6.4 SendSmtpMailMessage.Body
- **Current emission authority**: Deterministic generator `gen_send_smtp_mail` (deterministic-generators.ts:555) emits `Body=""` as attribute (line 559: `args.body || args.Body || ""`). `repairMailBodyAttributes` (required-property-enforcer.ts:1960, called from package-assembler.ts:8334) converts to child element but preserves empty value.
- **Enforcer authority**: None — no `FALLBACK_ELIGIBLE_POLICIES` entry, no `CONTEXT_DERIVED_FALLBACK_REGISTRY` entry.
- **Gap**: No fallback authority. The `repairMailBodyAttributes` function addresses structural format (attribute → child element) but not content validity.
- **Where authority should live**: A `FALLBACK_ELIGIBLE_POLICIES` entry with `contextDerivedFallbackAllowed: true` and a `CONTEXT_DERIVED_FALLBACK_REGISTRY` entry should be added, providing a minimum-valid body expression (e.g., derived from the Subject or a placeholder string). This is consistent with the LogMessage.Message pattern.

### 6.5 QueryEntity.EntityType
- **Current emission authority**: No deterministic generator (absent from `GENERATOR_REGISTRY` in generator-registry.ts). Emitted by workflow-tree-assembler or rich xaml-generator with `EntityType=""`.
- **Enforcer authority**: `FALLBACK_ELIGIBLE_POLICIES` entry at line 252 with `fallbackAllowed: false`, `blockOnGenericDefault: true`. `contextDerivedFallbackAllowed` not set. No `CONTEXT_DERIVED_FALLBACK_REGISTRY` entry.
- **Gap**: Policy correctly blocks but provides no recovery. EntityType is inherently domain-specific (requires knowledge of Orchestrator entity schema name).
- **Where authority should live**: The blocking behavior is appropriate — EntityType cannot be meaningfully auto-generated. Authority must remain with spec/user input. The defect messaging should be enhanced to provide actionable remediation guidance (e.g., "EntityType must be specified in the workflow spec — valid values include Orchestrator entity type names").
