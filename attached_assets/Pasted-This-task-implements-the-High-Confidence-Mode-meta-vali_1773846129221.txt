This task implements the High Confidence Mode meta-validation 
layer for CannonBall's package generation pipeline. It builds 
on the hierarchical tree architecture and must only be started 
after that architecture is complete and producing packages with 
a measurable templateComplianceScore.

The meta-layer is not a re-generation. It is a targeted, 
narrow review that applies specific corrections to an already-
assembled package. It is designed to be fast, cheap, and 
optional — engaged either by user choice or by automatic 
detection of high-risk generation conditions.

PART 1 — Complexity scoring function
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create server/pipeline/confidence-scorer.ts

This module evaluates the risk profile of a generation 
request and returns a score from 0.0 to 1.0. Scores above 
the threshold trigger automatic meta-validation. The scoring 
function is deterministic — no LLM calls.

type ConfidenceSignal = {
  name: string
  weight: number      // contribution to total score
  triggered: boolean  // whether this signal fired
  reason: string      // human-readable explanation
}

type ConfidenceAssessment = {
  score: number                    // 0.0 = low risk, 1.0 = high risk
  autoEngageMetaLayer: boolean     // score >= AUTO_ENGAGE_THRESHOLD
  signals: ConfidenceSignal[]
  summary: string                  // one-line explanation for the UI
}

const AUTO_ENGAGE_THRESHOLD = 0.6

PROCESS COMPLEXITY SIGNALS (from WorkflowSpec and SDD):

workflowCount > 4
  weight: 0.20
  reason: "Process requires {N} workflows — complex multi-file 
  generation"

activityCount > 50
  weight: 0.15
  reason: "High activity count ({N}) increases nesting risk"

externalSystemCount > 2
  weight: 0.15
  reason: "Multiple external integrations ({N} systems) 
  increase API call complexity"

hasREFramework
  weight: 0.10
  reason: "REFramework pattern requires strict structural 
  compliance"

hasDocumentUnderstanding or hasAICenter
  weight: 0.10
  reason: "Advanced UiPath services require catalog entries 
  not in bootstrap set"

QUALITY RISK SIGNALS (from pipeline results):

templateComplianceScore < 0.9
  weight: 0.25
  reason: "Template compliance at {score} — {N} activities 
  deviated from catalog schemas"

catalogViolationCount > 0
  weight: 0.20
  reason: "{N} catalog violations detected in validator output"

uncataloguedActivityCount > 0
  weight: 0.15
  reason: "{N} activities in spec not present in bootstrap 
  catalog"

qualityGateWarningCount > 5
  weight: 0.10
  reason: "{N} quality gate warnings suggest generation 
  instability"

PIPELINE SIGNALS:

isFirstGeneration (idea has never been generated before)
  weight: 0.10
  reason: "First generation for this process — no prior 
  baseline to compare against"

priorGenerationHadStubs (previous package had stubs)
  weight: 0.20
  reason: "Previous generation produced stubs — high risk 
  of recurrence"

Export:
assessConfidence(spec: WorkflowSpec, pipelineResult: 
  PipelineResult, ideaHistory: IdeaGenerationHistory
): ConfidenceAssessment

PART 2 — Targeted review prompt
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create server/pipeline/meta-validator.ts

This module constructs and executes the targeted review 
call. The review is narrow and structured — it checks 
specific known failure modes, not the entire XAML. This 
keeps the call fast and cheap.

REVIEW PROMPT STRUCTURE

The prompt has three sections:

SECTION 1 — ROLE (fixed)

"You are a UiPath XAML validator. Your job is to find 
specific categories of errors in generated XAML and 
return a structured list of corrections.

You are NOT rewriting the XAML. You are NOT reviewing 
style, naming, or logic. You are ONLY checking for the 
specific error categories listed below.

For each error found, return a structured correction 
object. If no errors are found in a category, return 
an empty array for that category.

Return ONLY a valid JSON object matching the 
CorrectionSet type. No explanatory text, no markdown, 
no code fences."

SECTION 2 — ERROR CATEGORIES TO CHECK

Inject only the categories relevant to the signals 
that triggered the meta-layer. Do not inject all 
categories on every call — scope the review to what 
the confidence scorer flagged.

Category definitions:

ENUM_VIOLATIONS
"Check every ui:LogMessage element. The Level attribute 
must be exactly one of: Trace, Info, Warn, Error, Fatal.
Report any other value as a correction."

NESTED_ARGUMENTS  
"Check every InArgument and OutArgument element. If an 
InArgument or OutArgument element's only child is another 
InArgument or OutArgument element of the same type, report 
it as a nested argument correction."

UNDECLARED_VARIABLES
"Check every variable name referenced inside square 
brackets in any OutArgument expression. If the variable 
name does not appear in the enclosing Sequence.Variables 
block, report it as an undeclared variable correction. 
Provide the inferred type based on naming convention:
str_ = x:String, b_ = x:Boolean, int_ = x:Int32, 
sec_ = s:SecureString, all others = x:Object"

FLAT_STRUCTURE
"Check every TryCatch element. Its Try body must contain 
at least one non-LogMessage activity. If the Try body 
contains only a LogMessage or is empty, and sibling 
elements immediately after the TryCatch appear to be 
the activities that should be inside it, report a 
nesting correction."

MISSING_PROPERTIES
"Check ui:SendSmtpMailMessage elements. Every instance 
must have a Body attribute. Report any missing Body 
as a correction.
Check ui:GetAsset elements. The output property must 
be AssetValue, not Value. Report any incorrect property 
name as a correction."

LITERAL_EXPRESSIONS
"Check all activity attributes that reference variables. 
Variable references must be wrapped in square brackets 
e.g. [variableName] not plain text. Report any 
unbracketed variable reference as a correction."

SECTION 3 — XAML INPUT

Inject the generated XAML files as named blocks:
"WORKFLOW: Main.xaml
{xaml content}

WORKFLOW: InitAllSettings.xaml  
{xaml content}"

Also inject the variable declarations from the 
WorkflowSpec so the validator knows what is declared:
"DECLARED VARIABLES:
{variable list from WorkflowSpec.variables}"

CORRECTION OUTPUT TYPE

The LLM must return a JSON object of this type:

type Correction = {
  workflow: string        // filename e.g. "Main.xaml"
  line?: number           // approximate line number if known
  category: string        // which error category
  description: string     // what is wrong
  currentValue: string    // the current incorrect XML/value
  correctedValue: string  // the corrected XML/value
  confidence: 'high' | 'medium' | 'low'  // how certain
}

type CorrectionSet = {
  corrections: Correction[]
  checkedCategories: string[]
  totalIssuesFound: number
}

COST CONTROLS

Limit the XAML input to 8,000 tokens maximum. If the 
generated XAML exceeds this limit, truncate to the 
first 8,000 tokens and add a note in the prompt: 
"Note: XAML truncated at 8,000 tokens. Focus review 
on the visible content only."

Set max_tokens on the API call to 2,000. Corrections 
are short structured objects — a complete correction 
set should never exceed this.

Use claude-haiku for this call, not claude-sonnet. 
The task is structured extraction, not reasoning. 
Haiku is faster and significantly cheaper for this 
use case. If Haiku is unavailable, fall back to 
the configured generation model.

PART 3 — Correction application layer
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Create server/pipeline/correction-applier.ts

This module applies the corrections returned by the 
meta-validator to the generated XAML files. All 
corrections are applied programmatically — no LLM 
involvement in the application step.

CORRECTION APPLICATION RULES

Only apply corrections with confidence: 'high' or 
'medium'. Discard confidence: 'low' corrections — 
log them as warnings but do not apply them.

For ENUM_VIOLATIONS:
  String replace Level="{wrongValue}" with 
  Level="{correctedValue}" scoped to the specific 
  workflow file.

For NESTED_ARGUMENTS:
  Use the collapseNestedArguments XML function 
  (already implemented) — re-run it as a targeted 
  pass on the affected workflow.

For UNDECLARED_VARIABLES:
  Add the variable declaration to Sequence.Variables 
  in the specified workflow using the variable 
  declaration template. Use the inferred type from 
  the correction object.

For FLAT_STRUCTURE:
  Log the correction as a warning but do not apply 
  automatically. Flat structure corrections require 
  structural XAML surgery that is too risky to apply 
  programmatically without validation. Surface as a 
  READY_WITH_WARNINGS condition instead.

For MISSING_PROPERTIES:
  Add the missing attribute to the specified activity 
  element using string replacement scoped to the 
  workflow file and approximate line number.

For LITERAL_EXPRESSIONS:
  Wrap the unbracketed value in square brackets using 
  targeted string replacement.

AFTER APPLYING CORRECTIONS

Re-run the catalog validator on the corrected XAML.
Re-run the quality gate.
If the corrected package now passes cleanly, set 
status to READY.
If warnings remain, set status to READY_WITH_WARNINGS 
with the remaining corrections logged.

Log a correction summary to the pipeline result:
correctionsApplied: number
correctionsSkipped: number  // low confidence
correctionsFailed: number   // applied but broke something
metaValidationDurationMs: number
metaValidationCost: number  // estimated token cost

Add High Confidence Mode to the CannonBall settings 
layer.

USER PREFERENCE OPTIONS (three modes):

"Auto" (default)
  Use the confidence scorer to decide. Engage meta-
  validation when score >= AUTO_ENGAGE_THRESHOLD.
  Label: "Smart — validates automatically for complex 
  processes"

"Always On"
  Always run meta-validation regardless of score.
  Label: "High Confidence — slower but maximises 
  first-time-right rate"

"Off"
  Never run meta-validation.
  Label: "Fast — standard generation, no extended 
  validation"

Store this preference per-user in the existing 
settings store.

UI PLACEMENT

Add to the idea detail page, near the Generate 
Package button. Show as a small toggle/selector 
with a tooltip explaining the tradeoff:

"High Confidence Mode runs an additional validation 
pass on generated packages. Adds ~10-15 seconds 
but catches more errors before you open in Studio."

When the mode is "Auto", show the current confidence 
assessment after generation starts:
"Complex process detected (score: 0.74) — running 
extended validation"

When mode is "Always On":
"Extended validation enabled"
