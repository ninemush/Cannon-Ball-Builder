Mail Family Lock and Narrative Container Elimination

What & Why
The latest runs still show the same dominant mail-send failures even after catalog filter alignment:
- Send SMTP Mail Message missing required Body
- UiPath.Mail.Activities discovered or aligned unexpectedly
- Gmail intent present in the structured spec, but downstream artifacts still drift into generic mail/SMTP-family output

The structured spec already contains enough evidence that the issue is not simply lack of LLM property knowledge. In the same logical send cluster, the spec currently contains both:
- a concrete GmailSendMessage step with To, Subject, and Body
- a narrative TryCatch representation describing the same Gmail send operation in plain-text properties

This allows the lowering path to preserve multiple competing representations of the same logical send operation, which can then drift into mixed Gmail/SMTP artifacts, lose required properties like Body, or introduce late package discovery for UiPath.Mail.Activities.

This task introduces a strict family lock for logical mail-send clusters and eliminates narrative container leakage before emission.

Goals
1. Ensure each logical mail-send cluster lowers into exactly one concrete mail family.
2. Prevent Gmail-intent clusters from producing SMTP or generic mail artifacts unless explicitly designed and fully contracted.
3. Eliminate prose/narrative TryCatch send descriptions from executable lowering unless they are converted into real container nodes.
4. Preserve required properties like Body through lowering for the winning family.
5. Fail early with structured diagnostics if a mail-send cluster is ambiguous or cannot be lowered cleanly.

Done looks like
- A logical send cluster is identified before XAML emission.
- Each send cluster chooses exactly one winning mail family:
  - Gmail
  - SMTP
  - Outlook
  - or explicit package-fatal rejection if ambiguous
- If a cluster contains both a concrete GmailSendMessage step and a narrative Try/Catch description of the same send, the pipeline either:
  - lowers them into one canonical executable representation with a real TryCatch container and a concrete Gmail child activity
  - or rejects the cluster before emission
- No Gmail-intent cluster produces SMTP-family artifacts unless SMTP was explicitly chosen and fully contracted.
- If Body exists in the structured spec for the winning family, it is preserved through lowering and not lost before emission.
- No placeholder injection occurs for rejected mail clusters.
- A structured diagnostics artifact shows cluster-level family selection and rejection reasons.

Out of scope
- Broad catalog/filter alignment
- General parity/freeze changes
- Full decomposition redesign
- UI-only reporting changes
- Deep business-logic invention

Tasks

1. Add logical send-cluster detection
Group related steps that represent one mail-send operation, including:
- concrete send steps
- adjacent retry wrappers
- adjacent TryCatch narrative wrappers
- nearby logging/catch steps clearly tied to the same send intent

2. Add strict mail-family lock
For each detected send cluster, choose exactly one mail family before emission:
- Gmail
- SMTP
- Outlook
If signals conflict across families, do not emit XAML for that cluster. Emit structured diagnostics and fail package mode / allow DHG remediation.

3. Eliminate narrative container leakage
If a cluster contains narrative container properties such as:
- Try = "GmailSendMessage(...)"
- Catches = "Exception -> ..."
- Finally = "None"
these must not pass downstream as literal executable properties.
They must be either:
- lowered into a real TryCatch container with concrete child activities for the winning family
- or rejected before emission

4. Preserve required properties through family lowering
If the winning family is Gmail and the source cluster contains Body/Subject/To in structured form, ensure the lowered concrete activity instance preserves them.
If a required property is missing at cluster resolution time, fail before emission rather than allowing downstream placeholder injection or family drift.

5. Block cross-family drift
Once a send cluster is locked to Gmail, no SMTP-family activity, package, or fallback artifact may be introduced for that same logical cluster during later lowering or assembly.
Add guardrails so downstream stages cannot re-map the cluster into a different mail family.

6. Emit cluster-level diagnostics
Add:
`mailFamilyLockDiagnostics`
with per cluster:
- workflow/file
- clusterId
- detectedIntent
- candidateFamilies
- selectedFamily
- concreteActivityType
- concretePackage
- requiredPropertiesSeen
- missingRequiredProperties
- narrativeRepresentationsRejected
- packageFatal
- remediationHint

Also include summary counts:
- totalMailClusters
- totalLockedSuccessfully
- totalRejectedForAmbiguity
- totalRejectedForMissingContract
- totalNarrativeContainerRejections
- totalCrossFamilyDriftBlocked

7. Integrate with package completeness policy
Wire rejected or incomplete mail clusters into packageCompletenessViolations so package mode fails early and honestly.

8. Add regression tests
Cover:
- concrete Gmail send with Body/Subject/To lowers to Gmail only
- Gmail cluster with narrative TryCatch wrapper is either correctly lowered to real TryCatch + Gmail child or rejected
- Gmail cluster never emits SMTP artifacts
- missing Body in the winning family is rejected before emission
- UiPath.Mail.Activities is not introduced for a Gmail-only cluster
- no placeholder injection occurs for rejected mail clusters
