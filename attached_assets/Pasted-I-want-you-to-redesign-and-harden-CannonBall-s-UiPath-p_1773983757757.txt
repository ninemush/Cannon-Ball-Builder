I want you to redesign and harden CannonBall’s UiPath package generation pipeline so it becomes systemic, upstream, and reliable across every idea in the app.

Context:
CannonBall takes a natural-language automation idea, generates the AS-IS process map, generates the TO-BE process based on available UiPath services, then generates the PDD, then SDD, then generates a UiPath package, and finally deploys it directly to UiPath. Users must also be able to download the package and open it in UiPath Studio.

Primary goal:
Get CannonBall’s package generation to produce near-executable, enterprise-grade output with minimal developer work, as close as possible to what a strong Sr UiPath Developer or Solution Architect would produce for production deployment.

Important product constraints:
- I do NOT want patch-style fixes.
- I want systemic, architecture-level improvements that affect every idea.
- I do NOT want the pipeline to fail the entire build just because some parts are imperfect.
- If some workflows or parts are invalid, unsafe, or incomplete, the system should replace only those specific parts with safe executable stubs or deterministic fallbacks.
- The final package should still be:
  - deployable directly from CannonBall to UiPath
  - downloadable from CannonBall
  - openable in UiPath Studio
- The DHG (Developer Handoff Guide) should clearly document every downgraded/stubbed/replaced/auto-repaired component and the exact remaining developer work.
- UX is critical. If users frequently end with “no package generated,” trust will collapse and CannonBall will fail as a product.

Current architecture to preserve and improve:
- There is already a canonical package pipeline.
- There is already an activity registry/catalog and policy layer.
- There is already a quality gate.
- There is already a meta-validator / review-and-repair capability.
- There is already fallback stub generation.
- There is already DHG generation.
- There are already baseline/full generation concepts.
- The fixes must build on this architecture, not replace it with ad hoc logic.

Architectural target:
I want package generation redesigned around a progressive convergence model with one canonical output contract.

Target convergence ladder:
1. Enterprise Attempt
- Try to generate the highest-quality, near-production implementation first.
- Use deterministic scaffolds and canonical workflow structures.
- Constrain LLM use to bounded enrichment tasks, not unconstrained package structure generation.
- Prefer registry-backed activities, canonical arguments, canonical properties, and deterministic packaging.

2. Auto-Repair + Targeted Replacement
- Run analyzer, meta-validator, normalization, and quality gate on each workflow.
- If a workflow has invalid or unsafe parts, do NOT fail the whole package.
- Replace only the smallest failing unit possible:
  - ideally a step/activity
  - if needed, a sequence
  - only as a last resort, the workflow
- Preserve all valid upstream/downstream structure where possible.
- Keep the package executable and coherent.

3. Guaranteed Openable / Deployable Fallback
- If enterprise-level implementation cannot be achieved for some parts, still ship a valid package.
- The final fallback package must remain deployable and Studio-openable.
- All unresolved gaps must be surfaced in the DHG clearly and precisely.
- Only truly catastrophic package-level corruption should block output.

Very important design principle:
Separate package viability from workflow excellence.
- Package viability = the .nupkg is structurally valid, deployable from CannonBall, downloadable, and openable in Studio.
- Workflow excellence = near-executable Sr Dev / SA quality implementation.
- Workflow excellence should be optimized for, but should not normally be allowed to kill package viability.

Critical current fail-closed paths that must be fixed:
1. Full implementation mode can still throw and kill the build after quality gate failure instead of degrading gracefully.
- I want this changed so that if full_implementation cannot converge after repair, the pipeline automatically retries/downgrades to baseline_openable and reports the downgrade clearly.

2. AI enrichment failure path must not kill package generation.
- If the LLM enrichment step fails, returns malformed output, or produces unusable workflow specs, I want CannonBall to fall back to deterministic scaffold generation directly from the process map + SDD context.

3. Empty-package / no-buffer outcome must be treated as an architectural failure path to eliminate.
- The system should almost never end in “no package generated.”
- If the package cannot be near-executable, it should still converge to a valid fallback package instead.

Specific implementation direction:
- Add an explicit successful degraded state such as FALLBACK_READY in addition to READY and READY_WITH_WARNINGS.
- Treat baseline_openable as a first-class reliable delivery mode, not a weak backup.
- Automatically downgrade from full_implementation to baseline_openable if quality gate remediation still cannot converge.
- Feed all auto-fix actions, replacements, stub insertions, and downgrade decisions into the DHG.
- Improve survivability granularity:
  - today per-workflow fallback exists
  - next I want per-activity or per-sequence replacement where possible
- Enforce a cleaner deterministic boundary:
  - the LLM may propose workflow tree structure and business logic intent
  - activity properties, namespaces, package versions, and canonical structural rules must come from the activity catalog / registry / deterministic assembler
  - strip or normalize non-catalog properties before they reach the quality gate
- The package should converge to one of:
  - READY
  - READY_WITH_WARNINGS
  - FALLBACK_READY
- Total package failure should be reserved only for truly catastrophic structural corruption.

DHG expectations:
The DHG must become the truth layer for degraded output. It should explicitly report:
- what was fully generated
- what was auto-repaired
- what was downgraded
- what was stubbed
- why each downgrade/stub happened
- what exact developer action is needed next
