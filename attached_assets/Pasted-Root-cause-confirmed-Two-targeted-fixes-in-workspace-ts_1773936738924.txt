Root cause confirmed. Two targeted 
fixes in workspace.tsx only.

FIX 1 — Reset isGeneratingDocRef in 
the done handler

The done handler inside handleSSEEvent 
(lines 1035-1043) does not reset 
isGeneratingDocRef. When a generation 
completes via the done event, the ref 
stays true until the finally block 
runs. But the finally block is guarded 
by isOwnGeneration — if anything 
causes that check to fail, the ref 
never gets reset.

Add an explicit reset to the done 
handler:

  if (data.done) {
    setStreamingMsg((prev) =>
      prev ? { ...prev, isStreaming: false } : prev
    )
    setDocProgressSection("")
    setClassifiedIntent("")
    setLiveStatus("")
    setTimeout(() => 
      setMetaValidationChipStatus("ready"), 
      5000
    )
    // ADD THESE TWO LINES:
    isGeneratingDocRef.current = false
    generatingDocTypeRef.current = ""
  }

This ensures the ref is reset 
immediately when the server sends 
done, not only when the finally 
block runs.

FIX 2 — Remove the stale-ref guard 
from docProgress.started

Line 1061's guard:
  if (data.docProgress.started && 
      !isGeneratingDocRef.current)

Is the direct cause of the SDD 
streaming not triggering after a 
UiPath run. The guard was intended 
to prevent duplicate startDocStreaming 
calls but it blocks legitimate new 
generation starts when the ref is 
stale.

The guard is unnecessary because 
startDocStreaming already handles 
re-entry safely — it increments 
docGenIdRef.current, clears all 
streaming state, and resets the 
elapsed timer. Calling it again 
for a new document type is correct 
behaviour, not a problem to prevent.

Change line 1061 from:
  if (data.docProgress.started && 
      !isGeneratingDocRef.current) {
    startDocStreaming(...)

To:
  if (data.docProgress.started) {
    startDocStreaming(...)

Apply the same change at line 985 
for the tag-based detection path:
  if (docTagMatch && 
      !isGeneratingDocRef.current) {

Change to:
  if (docTagMatch) {

Both detection paths must allow 
startDocStreaming to be called 
regardless of current ref state. 
The ref reset in Fix 1 and 
startDocStreaming's own internal 
reset are sufficient to handle 
re-entry correctly.
