WeatherAndTravelNotification baseline feedback

Purpose:
This file describes a locally repaired baseline project that parses cleanly as XML and is intended to be Studio-openable as a minimal flat UiPath process scaffold.

What was changed locally:
1. Replaced malformed generated XAML with minimal valid root-level workflows only.
2. Removed all broken references to Workflows\ subfolder.
3. Removed scaffold-only artifacts such as screenshot logging patterns and diagnostic add-log-fields patterns.
4. Removed malformed attributes and invalid XML syntax.
5. Reduced project.json dependencies to a minimal baseline:
   - UiPath.System.Activities [23.10.3]
6. Corrected project profile spelling from Developement to Development.
7. Kept the project flat:
   - Main.xaml
   - InitAllSettings.xaml
   - GetWeather.xaml
   - GetC2CStatus.xaml
   - GetDistrictLine.xaml
   - BuildEmailBody.xaml
   - SendEmail.xaml
   - project.json
   - project.uiproj
   - entry-points.json

Key differences from CannonBall output:
1. All InvokeWorkflowFile paths are root-relative filenames only.
   Example:
   - Good: GetWeather.xaml
   - Bad: Workflows\GetWeather.xaml

2. No hidden REFramework assumptions.
   - No Workflows folder requirement
   - No screenshot capture scaffolding
   - No AddLogFields scaffolding
   - No placeholder tokens

3. No unsupported activity schema emitted.
   - The broken generated package used activityType values and properties that its own quality gate later rejected.
   - A generator must only emit activity names and properties that its validator recognizes as valid.

4. Minimal dependency discipline.
   - The broken generated package injected UiPath.UIAutomation.Activities, UiPath.Testing.Activities, UiPath.Mail.Activities, UiPath.Credentials.Activities, and Newtonsoft.Json without proving those were required by a valid final package.
   - A Studio-openable baseline should not include extra dependencies unless they are required by surviving activities.

5. Self-consistency matters more than ambition.
   - If the archive contains GetWeather.xaml at root, then Main.xaml must reference GetWeather.xaml at root.
   - The previous generator emitted a package model that was self-inconsistent before packaging even finished.

Minimum Studio-openable acceptance gate learned from this baseline:
1. Every XAML must be valid XML.
2. Every referenced workflow path must exist exactly as packaged.
3. Every emitted activity must be known to the declared dependency set.
4. Every variable and argument must be declared before use.
5. No placeholder or scaffold tokens may remain.
6. Dependency versions must be compatible with the declared Studio/Windows target.
7. Packaged file list must exactly match the reviewed raw manifest.

What CannonBall should learn:
1. Flat-file generation is safer than scaffold injection for simple linear API automations.
2. The generator and validator must use the same activity schema.
3. Package metadata must be internally consistent before packaging starts.
4. A minimal valid baseline is more useful than a richer but invalid scaffold.
5. If the package cannot yet implement full business logic safely, generate a clean openable baseline first and let Studio be the authoring surface for the next layer of work.
