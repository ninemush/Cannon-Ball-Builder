Critical Activity Required-Property Parity and Stage-Ordering Fix

What & Why
Recent benchmark runs still fail on required properties such as:
- GmailSendMessage.Body
- UpdateEntity.EntityObject

Code inspection shows the pipeline already has multiple required-property mechanisms:
- required-property-enforcer.ts
- post-assembly required-property enforcement in package-assembler.ts
- missing-required-property checks in uipath-quality-gate.ts

The problem is likely not missing capability. The problem is that these paths are not aligned on:
- when a property is considered satisfied
- whether child-element form counts as satisfied
- whether fallback/handoff bindings emitted later in the pipeline are recognized by earlier or parallel validators
- which stage is authoritative

For example, the Gmail send template in workflow-tree-assembler.ts already emits a .Body child element with a fallback str_EmailBody path, yet the package still reports Gmail Body as missing. That strongly suggests a parity / stage-ordering bug rather than a missing template implementation.

This task must not build a new required-property framework. It must align the existing enforcement and validation paths so they evaluate the same final property truth.

Goals
1. Make required-property satisfaction consistent across enforcer, post-assembly enforcement, and quality gate.
2. Ensure child-element properties like GmailSendMessage.Body and UpdateEntity.EntityObject are recognized as satisfied when they are present in valid final form.
3. Eliminate false missing-required-property defects caused by stage-ordering mismatch.
4. Patch only the specific inconsistent branches proven by benchmark evidence.

Done looks like
- GmailSendMessage.Body is no longer reported missing when the final emitted activity contains a valid .Body child element.
- UpdateEntity.EntityObject is no longer reported missing when the final emitted activity contains a valid .EntityObject child element.
- required-property-enforcer, package-assembler post-assembly checks, and uipath-quality-gate agree on whether a required property is satisfied.
- The benchmark BirthdayGreetingsV26 case no longer fails on false missing Body / EntityObject defects caused by stage mismatch.
- If a property is truly missing, the defect still appears honestly.

Out of scope
- Rebuilding required-property enforcement from scratch
- Broad mail-family redesign
- Auto-wiring framework redesign
- Dependency-family redesign
- Business-value invention

Tasks

1. Trace the live required-property decision chain for the benchmark failures
For:
- GmailSendMessage.Body
- UpdateEntity.EntityObject

Trace, on the active path:
- whether the property exists in the source node
- whether it exists after lowering
- whether it exists after required-property-enforcer
- whether it exists after template resolution
- whether it exists after compliance/normalization
- whether it exists in final archive-bound XAML
- which subsystem first declares it missing

2. Identify the authoritative required-property truth point
Decide which stage is authoritative for “property satisfied” for critical activities:
- pre-template source property
- post-template emitted child/attribute form
- post-compliance normalized XAML

Use the benchmark evidence to choose the authoritative stage. Do not guess.

3. Align the existing checks to the same truth
Patch existing logic so:
- required-property-enforcer
- package-assembler post-assembly required-property checks
- uipath-quality-gate missing-required-property checks

all evaluate required properties against the same authoritative representation.

4. Explicitly support child-element required properties
Ensure the aligned checks correctly recognize child-element required properties, at minimum:
- GmailSendMessage.Body
- UpdateEntity.EntityObject
and do not treat them as missing merely because they are not present as attributes.

5. Emit lightweight parity diagnostics
Add or extend one existing diagnostics surface with enough information to show:
- which subsystem flagged the property missing
- what the property shape was at that point
- what the authoritative final shape was
Do not create a heavy parallel diagnostics framework unless needed.

6. Evidence-gated patching only
No speculative fixes.
Patch only the branch proven by benchmark/runtime evidence.
If the root cause is:
- enforcer timing
- post-assembly enforcement
- quality-gate shape recognition
- compliance rewriting
patch that exact branch only.

7. Add focused regression tests
Cover:
- GmailSendMessage.Body present as child element is recognized as satisfied
- UpdateEntity.EntityObject present as child element is recognized as satisfied
- quality gate and post-assembly enforcement agree on those cases
- truly missing required properties still fail honestly
- no new parallel required-property system is introduced
