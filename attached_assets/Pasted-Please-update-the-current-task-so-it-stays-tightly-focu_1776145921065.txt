### 1) `compliancePass()` side effect creates duplicate `xamlEntries`

Problem:

* `Process.xaml` is generated once, then later rebuilt during spec-aware wiring.
* Some call sites invoke `compliancePass(...)` inline inside `syncXamlWrite(...)`.
* Because JavaScript evaluates arguments before the outer function call, `compliancePass(...)` mutates tracking state first and pushes a new entry into `xamlEntries`.
* `syncXamlWrite(...)` then updates an existing entry but does not eliminate the newly pushed orphaned duplicate.
* This creates duplicate tracked entries for the same XAML file, especially `Process.xaml`.

Required fix:

* Audit **all inline `compliancePass(...)` calls passed directly into `syncXamlWrite(...)`**.
* In those cases, suppress tracking side effects by using the supported `skipTracking` option.
* `syncXamlWrite(...)` must be the single owner of tracking in those code paths.
* Keep the fix surgical. Do not redesign the broader compliance architecture.

### 2) Duplicate-file remediation is treating bookkeeping duplication like XML corruption

Problem:

* The gap analyzer detects duplicate XAML entries and emits `duplicate-file`.
* Current remediation treats that as a severe validation failure and replaces the file with a stub.
* For `Process.xaml`, that destroys the invocation chain and makes spec workflows unreachable.
* A duplicate bookkeeping record is not the same as malformed XML.

Required fix:

* Handle `duplicate-file` separately from actual XML/content corruption categories.
* For duplicates, deduplicate entries rather than stubbing the file.
* Keep the latest valid content and remove extras.
* Stub remediation must remain reserved for true corruption categories such as malformed XML, well-formedness failures, malformed quotes, and similar content-level breakage.

### 3) `GetCredential` object-as-string crash in workflow assembly

Problem:

* `Username` and `Password` can arrive as `ValueIntent`-style objects instead of raw strings.
* Current logic later assumes string semantics and calls `.replace(...)`.
* This crashes assembly of workflows like `CoupaLogin`, causing fallback to structural stubs.

Required fix:

* In both `crossCheckGetCredentialVariableTypes()` and the workflow tree assemblerâ€™s `GetCredential` handling, resolve object-shaped values to scalar strings before further processing.
* Reuse the same scalar-string resolution helper already used by deterministic generators.
* Guard string operations with type checks or explicit coercion after resolution.
* Never call `.replace(...)` on unresolved objects.

### 4) Safety net for critical infrastructure files

Problem:

* If `Main.xaml` or `Process.xaml` is ever stubbed, the invocation chain can be completely severed.

Required fix:

* Before stubbing critical orchestrator files such as `Main.xaml` or `Process.xaml`, extract any existing `InvokeWorkflowFile` references from the original content when possible.
* Preserve those invoke relationships in the generated stub as a defense-in-depth safeguard.
* This is not the primary fix. It is a protective fallback.
