The package generation is failing because the tree 
enricher's maxTokens limit of 8192 is too low for 
the hierarchical WorkflowSpec JSON output. The AI 
is consistently truncating at ~32-33K characters 
which produces malformed JSON on every attempt.

THREE CHANGES REQUIRED:

CHANGE 1 — Increase maxTokens for the tree enricher
In the tree enricher (enrichWithAITree, line ~541), 
increase maxTokens from 8192 to 16384.

Do NOT change the legacy flat enricher — its 8192 
limit is appropriate for its simpler output format.

CHANGE 2 — Implement chunked generation as fallback
If the response still fails JSON validation after 
increasing the limit, do not retry the full 
generation. Instead fall back to a chunked strategy:

Split the process nodes into batches of 4 nodes 
maximum. For each batch:
  - Call the tree enricher with just those 4 nodes
  - Validate the JSON response
  - If valid, store the partial WorkflowSpec
  
After all batches complete, merge the partial 
WorkflowSpec objects:
  - Combine all variables arrays (deduplicate by name)
  - Combine all root sequence children in order
  - Use the first batch's workflow metadata

The chunked fallback only engages when the full 
generation fails — it is not the default path. 
Most processes with fewer than 8 nodes will succeed 
within the 16384 limit.

CHANGE 3 — Fix the JSON repair logic
The current JSON repair on truncated responses 
(the retry with error context) is not effective 
because the fundamental problem is truncation, not 
malformed generation. Replace the current retry 
strategy:

Current (wrong approach):
  Generate → fails → retry with error context

Replace with:
  Generate → fails JSON validation → attempt 
  auto-repair → if repair fails → chunked fallback

The auto-repair should attempt to close unclosed 
structures using a simple recursive closer:
  - Count unclosed { and [ brackets
  - Append the correct closing sequence
  - Re-validate
  - If valid after repair, use the repaired JSON
  - If still invalid, fall back to chunked generation

This handles the common truncation pattern where 
the JSON is complete except for missing closing 
brackets at the end.
