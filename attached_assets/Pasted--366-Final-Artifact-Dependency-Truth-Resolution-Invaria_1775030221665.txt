#366 - Final Artifact Dependency Truth & Resolution Invariants

What & Why
Task #365 improved the pipeline materially, but the latest BirthdayGreetingsV12 artifact still proves that final artifact truth is not yet enforced tightly enough. The package opens in Studio with unresolved activity/type errors, duplicate bracket-named workflows still survive into the package, and the latest forensic findings suggest pre-emission validation may be bypassed or non-authoritative for some flows.

The next bottleneck is no longer broad emission hardening in general. It is final artifact truth:
- the final emitted XAML must agree with project.json dependencies
- the final emitted XAML headers must contain the correct xmlns / AssemblyReference / namespace imports for the activities actually used
- canonical duplicate workflows must not survive into the final package
- pre-emission validation must actually execute and be observable
- complex expressions must not be “helpfully normalized” into broken VB

This task establishes those invariants.

Done looks like
The final packaged artifact contains zero canonical duplicate workflows after final-write validation
No emitted activity backed by a declared dependency remains unresolved in Studio due to missing xmlns / AssemblyReference / namespace-import wiring
Final artifact dependency/header reconciliation passes for every XAML file before packaging
Pre-emission validation reports non-zero activity/workflow coverage for non-empty builds; zero-coverage is treated as a blocking pipeline health failure
GetAsset, GetCredential, RetryScope/ShouldRetry, and other emitted activity families either resolve correctly in the final artifact or block the build with a precise diagnostic
Complex expressions that cannot be safely normalized no longer get mangled into invalid VB; they either pass through a conservative validity gate or fail with a blocking diagnostic
BirthdayGreetingsV12 no longer emits bracket-named duplicate files like [GetTodayBirthdays.xaml].xaml into the final package

Out of scope
Broad scanner-based undeclared-variable auto-declaration
Defaulting unknown identifiers to x:String
New generic late-stage repair passes
Broad regex-based expression salvage
Large stage-ownership refactors
Broad prompt redesign
Using DHG readiness percentage itself as the task acceptance criterion

Tasks

1. Final artifact dependency/header reconciliation
Add a dedicated final reconciliation stage after normalization and before archive/package write that:
- scans each final emitted XAML file for actual activity tags/types/prefixes used
- resolves each to the authoritative package / namespace / assembly mapping
- verifies the same file’s XAML header contains the required xmlns declarations, AssemblyReference entries, and namespace imports
- verifies project.json contains the required package dependency
If a required mapping is known and missing, inject it deterministically.
If the mapping is unknown or ambiguous, emit a blocking diagnostic instead of silently packaging a Studio-broken artifact.

2. Final canonical dedup invariant
Keep pre-assembly dedup, but add a second hard invariant check immediately before package/archive write:
- canonicalize every final workflow/file name by stripping brackets, quotes, .xaml suffixes, whitespace, and case differences
- if multiple final files collapse to the same canonical name, block packaging
- surface the exact colliding names in the diagnostic
This must apply to deferredWrites, xamlEntries, and any final manifest/archive list.

3. Pre-emission validation health invariant
Diagnose why spec/pre-emission validation can report zero coverage on a non-empty build.
Add a pipeline health invariant:
- if workflows/activities were generated but pre-emission validation reports zero checked items, treat that as a blocking failure or explicit degraded-health state
- log validation coverage counts in the build summary and DHG context
This is not just observability — it should prevent the build from appearing healthy when the early-warning layer did not actually run.

4. Safe-by-default complex expression handling
Refine the expression path so that for complex expressions (e.g. lambdas, LINQ, nested calls, multiple operators):
- heuristic normalization is skipped
- the original expression text is preserved
- the expression is allowed through only if it passes a conservative validity check
- otherwise emit a blocking diagnostic with the exact expression and reason
Do NOT simply “emit as-received from the LLM” without validation.
Do NOT attempt broad regex salvage of complex expressions.

5. Activity family resolution checks for recurring failures
Add explicit resolution checks for the recurring emitted activity families that keep failing in Studio:
- GetAsset / GetCredential
- RetryScope / ShouldRetry
- Integration Service / GenAI related activities
- mail/data-service families if emitted
For each family, verify:
- package dependency
- xmlns prefix mapping
- assembly reference
- namespace import
- any known package-version-specific namespace assumptions
If the artifact cannot satisfy the contract for that family, block it rather than allowing unresolved activities into Studio.

Implementation notes / guardrails
- This should be a final artifact truth task, not another broad “fix-it-late” mutation stage.
- Inject only missing declarations that are authoritative and deterministic.
- If a dependency/type/header contract cannot be satisfied with high confidence, fail clearly.
- Keep Option A narrow; this task should sit after normalization as a final artifact invariant/reconciliation layer, not expand Option A into a generic fixer.
