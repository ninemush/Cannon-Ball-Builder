You are implementing **Task #518 - Systemic package generation quality fix** in CannonBall.

Your job is to make the package-generation pipeline more systemic and less symptom-driven. Do not make narrow local fixes that only address one observed run. Implement the changes below so they generalize across the app and reduce structural invalidity, stubs, and handoff-only outcomes.

## Objective

Fix the three systemic gaps now blocking package quality:

1. **CLI validation is currently a dead end**

   * UiPath CLI validation runs after iterative LLM correction, so CLI defects are logged but never acted on.
   * `cliDefectsToHealingInput()` already exists in `server/uipath-cli-validator.ts` but is not wired into the correction flow.
   * The iterative corrector currently only sees internal quality-gate violations.

2. **Emission approval is bottlenecked on namespace lookup**

   * `catalog/run-emission-triage.js` still checks namespace mappings by scraping hardcoded entries from `xaml-compliance.ts`.
   * The catalog service already exposes DLL-derived namespace data via `getAllPackageNamespaceEntries()`.
   * There are about **227 activities** with `propertiesComplete: true` and `emissionApproved: false` that are safe approval candidates blocked only by this namespace lookup gap.

3. **The AI still generates XAML too blindly**

   * The enrichment prompt gives schema and filtered catalog context, but no concrete XAML structural patterns.
   * We already have completed project XAML and Studio baselines. Use them to teach structural patterns, not business logic.

---

## Critical design constraints

### A. Do NOT run the iterative corrector twice

Do **not** implement “CLI runs after correction and then invoke a second correction pass.”

Reason:

* the iterative corrector verifies fixes using internal quality-gate revalidation
* CLI defect resolution is not visible to the internal QG
* this would trigger the corrector’s “revert if no improvement” logic and undo valid CLI-driven fixes
* it would also double token cost

### B. Reorder the pipeline instead

The correct design is:

1. run internal/meta validation
2. run **CLI analyze-only before iterative correction**
3. convert CLI defects with `cliDefectsToHealingInput()`
4. merge CLI-derived defects into the existing `qgViolationsForIterative`
5. run the existing iterative LLM correction **once**
6. rebuild outputs
7. run final CLI validation (`analyze + pack`) as the final verification gate
8. all downstream `cliValidationResult` consumers must use the **final post-correction result**

### C. Keep emission expansion safe

Only approve candidates that satisfy all of:

* `propertiesComplete: true`
* package namespace metadata exists from DLL extraction
* package/activity already satisfies the rest of triage safety conditions

Do **not** approve the incomplete pool.

### D. Keep XAML prompt additions compact

The enrichment stage has a 45-second timeout and already carries substantial context.
Add only **2-3 focused structural snippets**, under roughly **1000 tokens total**.
Do not paste long full workflows.
Do not add presentation/view-state clutter such as `sap:HintSize` or `sap2010:WorkflowViewState`.
