## Required refinements to #530

Keep #530 as the second implementation task, but **soften the hard guard**.

### 1. Replace hard package-fail behavior with repair-first localized degradation

The current idea of “throw on invalid x:Key” is too aggressive for CannonBall.

Update #530 so that when the generic invoke emitter resolves an argument binding key that is:

* null
* undefined
* empty
* whitespace-only
* equal to `Nothing`
* or otherwise not a valid callee argument name

the pipeline must do this in order:

#### Step 1: Deterministic repair first

Attempt to recover the intended key from:

* the callee contract
* spec intent / binding metadata
* a unique unambiguous argument match, if one exists

Preserve the original offending value and caller intent.

#### Step 2: If repair fails, degrade only the single invoke

Do **not** fail the whole package.

Instead:

* replace only the affected `InvokeWorkflowFile` activity with a localized `Comment + LogMessage` handoff stub
* preserve the surrounding workflow and control-flow structure
* preserve all unrelated activities and workflows
* preserve the DHG trace of original caller intent and the manual fix required

#### Step 3: Only hard-fail as a last resort

Hard-fail only if the broken invoke is in a truly non-localizable structural position and no safe local replacement is possible.

### 2. Keep an emission guard, but change its failure mode

The emission-time guard should still detect invalid resolved keys before writing XAML.

It must still emit a diagnostic containing:

* invoke display name
* target workflow
* offending value
* attempted key

But its default response must be:

* deterministic repair attempt
* then invoke-level degradation
* not whole-package failure

### 3. Strengthen validation when callee contract is known

When the callee contract is known at emission time:

* emitted `x:Key` must be validated against the declared callee argument set
* reject keyword / null / empty / unresolved keys before they reach XAML

### 4. Tighten regression expectations

Require assertions for:

* zero emitted `x:Key` values equal to `Nothing`
* zero `propertyName === "Nothing"`
* zero `targetArgument === "Nothing"`
* zero `referencedSymbol === "Nothing"`
* preserved invoke semantics for the affected workflows
* no whole-package failure caused solely by null-key invoke-binding defects

## Required refinements to #528

Keep #528 as the third task, but narrow and tighten it.

### 1. Refine the invariant

Change the invariant from:

> “no pipeline-generated artifact can cause a structurally_invalid verdict”

to:

> “no pipeline-generated artifact that is Studio-safe after sanitization and smallest-scope localized degradation can by itself cause a structurally_invalid verdict”

This distinction is critical.

### 2. Make provenance non-blanket

State explicitly:

* pipeline-generated provenance is **not** a blanket exemption
* if a pipeline-generated artifact still leaves malformed XML, broken activity structure, broken invoke contract, missing Main.xaml, or non-openable package state, it may still participate in a genuine structural failure outcome
* provenance-aware scoring must only soften **safe, intentional fallback artifacts**, not real structural breakage

### 3. Preserve artifact-origin tagging at creation time

Keep and strengthen this requirement:

* provenance must be attached **at the moment a fallback artifact is created**
* it must be carried through the pipeline into quality gate and final verdict logic
* retrospective string matching must **not** be the primary origin-classification method

### 4. Pre-compliance guard must become provenance-aware, but not blind

Refine the pre-compliance guard behavior so that:

* pipeline-generated safe fallback tokens do not by themselves flip `preComplianceGuardPassed` to false
* but genuinely unsafe or still-structural pipeline output may still do so
* the distinction must be based on provenance + Studio-safety after sanitization/local degradation, not mere token presence

### 5. Compliance-crash cascade softening must be bounded

Refine the “close the compliance-crash cascade” section so that:

* if compliance crash is attributable to a localized, provenance-tagged pipeline artifact, attempt smallest-scope recovery first
* only apply local degradation if the workflow can still be re-emitted as well-formed XML
* otherwise escalate normally

### 6. Add mixed-case tests

Require at least these cases:

* pipeline-fallback-only safe defects → must not produce `structurally_invalid`
* genuine structural defects → must still produce `structurally_invalid`
* mixed safe-fallback + genuine structural defects → must still produce `structurally_invalid`
