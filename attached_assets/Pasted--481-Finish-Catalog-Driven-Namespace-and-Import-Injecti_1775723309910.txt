#481 - Finish Catalog-Driven Namespace and Import Injection Authority

What & Why
The DLL-backed migration made the catalog the structural source of truth for package prefix, clrNamespace, and assembly metadata. Most catalog-first rewiring for namespace resolution is already in place. However, two critical derived maps still come only from the hardcoded PACKAGE_NAMESPACE_MAP in server/xaml/xaml-compliance.ts:

- PREFIX_TO_XMLNS
- CLR_NAMESPACE_TO_XAML_PREFIX

This means namespace injection and CLR-type-to-prefix resolution can still drift from catalog truth even when catalogService is loaded. The latest benchmark behavior remains consistent with this gap: both Coupa and Birthday still fail with namespace import / assembly reference injection issues.

This task must close the remaining delta by rebuilding those derived maps from catalog truth when available, with the hardcoded map preserved only as an explicit fallback. It must also add mismatch diagnostics so catalog-vs-fallback drift is no longer silent.

Done looks like
- PREFIX_TO_XMLNS uses catalog package namespace metadata when catalogService is loaded, with hardcoded fallback only when catalog data is unavailable.
- CLR_NAMESPACE_TO_XAML_PREFIX uses catalog package namespace metadata when catalogService is loaded, with hardcoded fallback only when catalog data is unavailable.
- injectMissingNamespaceDeclarations resolves missing xmlns declarations from catalog-derived PREFIX_TO_XMLNS when catalog data is available.
- normalizeVariableTypeAttr and any CLR-type-to-prefix resolution path use catalog-derived CLR namespace mapping when available.
- Catalog-vs-fallback namespace mismatches emit diagnostics instead of remaining silent.
- Coupa and Birthday benchmark runs materially reduce or eliminate the shared failure class:
  - failed to inject namespace imports
  - failed to inject assembly references
  where that failure is caused by stale hardcoded namespace mapping drift.

Out of scope
- Rebuilding getActivityPrefixStrict, resolvePackageNamespaceInfo, or collectUsedPackages if they are already catalog-first and working
- Creating new catalog accessors if existing ones already cover the needed namespace metadata
- Broad catalog or generator rewrites
- Required-property fallback policy changes
- Deterministic generator redesign
- Prompt engineering
- Full elimination of PACKAGE_NAMESPACE_MAP if a safe degraded-mode fallback is still needed

Design rules
Catalog first, fallback second
When catalogService is loaded, derived namespace maps must be built from catalog package namespace metadata first. The hardcoded PACKAGE_NAMESPACE_MAP may remain only as an explicit fallback.

No silent dual truth
If catalog-derived namespace data and fallback PACKAGE_NAMESPACE_MAP disagree for the same package, prefix, clrNamespace, or assembly, emit a structured diagnostic rather than silently choosing one.

No regression when catalog is unavailable
If catalogService is not loaded, preserve current fallback behavior so the pipeline still works in degraded mode.

Minimal-delta fix
Do not rework paths that are already catalog-first and functioning. Limit the task to the remaining stale derived maps and the related validation of injection paths.

Add mismatch diagnostics.

- packageId

- source selected

- injectMissingNamespaceDeclarations injects xmlns from catalog-derived data
