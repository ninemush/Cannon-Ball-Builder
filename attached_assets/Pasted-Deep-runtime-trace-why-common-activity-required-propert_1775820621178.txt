Deep runtime trace: why common-activity required-property fallbacks do not survive into final package output

What & Why
The current codebase already contains required-property fallback infrastructure for at least some of the exact activity/property pairs still failing in generated packages, especially common activities like:
- LogMessage.Message
- InvokeWorkflowFile.WorkflowFileName

Yet the latest generated packages still show these properties as missing in the final output.

Before implementing another fix, we need a surgical runtime trace to determine exactly where the intended fallback or minimum-valid-value path is being lost.

This is an investigation task, not an implementation task.

The goal is to answer, with code evidence, why the fallback logic exists but the final package still emits invalid common activities.

Questions to answer
1. For each of these failing activity/property pairs:
- LogMessage.Message
- InvokeWorkflowFile.WorkflowFileName
- ReadTextFile.File
- SendSmtpMailMessage.Body
- QueryEntity.EntityType

trace the full path from:
- spec / normalized spec
- workflow-tree-assembler or deterministic generator input
- required-property-enforcer
- any post-emission mutation
- final deferredWrites / xamlEntries content
- final emitted XAML in package assembly

2. For LogMessage.Message and InvokeWorkflowFile.WorkflowFileName specifically:
- does the required-property-enforcer actually see these activities?
- do activityType and propertyName match the fallback registry keys exactly?
- does tier 7 or context-derived fallback execute?
- if it executes, where is the fallback value written?
- does a later stage overwrite, strip, stub, or bypass that value?

3. For ReadTextFile.File, SendSmtpMailMessage.Body, and QueryEntity.EntityType:
- are these expected to be handled by generator logic, enforcer logic, or both?
- is there currently any minimum-valid fallback path at all?
- if not, where is the missing authority supposed to live?

4. Are the failing workflows bypassing the enforcer entirely because they:
- are stubbed before enforcement
- are emitted through a codepath that never calls the enforcer
- use activity names/aliases that do not match policy keys
- are mutated after enforcement using stale content
- are removed as unreachable before the corrected content survives

5. What is the earliest stage where each of the five failing required properties becomes invalid or absent?
