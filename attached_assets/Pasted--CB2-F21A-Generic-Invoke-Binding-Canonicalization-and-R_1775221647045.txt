# CB2-F21A - Generic Invoke Binding Canonicalization and Residual Expression Serialization Cleanup

What & Why
The latest package run shows that designer metadata exclusion is now partially working, but the package is still structurally invalid because workflow invocation serialization and expression-bearing property serialization are still emitted in non-canonical or invalid forms.

The dominant remaining defects are:
- leaked JSON-like expression payloads in executable properties
- invalid workflow invocation serialization
- pseudo-properties emitted as child-argument bindings
- mixed literal/expression syntax
- placeholders in required executable properties

The current workflow contract validator is now correctly excluding some designer metadata, but many remaining contract defects are downstream symptoms of bad serialization rather than true contract mismatches. In particular, InvokeWorkflowFile nodes are still being emitted with conflicting or non-supported binding shapes such as:
- argument blob plus per-argument pseudo-properties
- JSON-like payloads inside expression-bearing attributes
- malformed mixed literal/expression forms

This task fixes the serialization layer generically by canonicalizing workflow invocation bindings and cleaning up residual JSON-like expression payloads across expression-bearing properties.

Goals
1. Emit one valid canonical workflow invocation binding form.
2. Stop serializing child workflow arguments as pseudo-properties on invoke activities.
3. Resolve or preserve residual JSON-like expression payloads in a deterministic, generic way.
4. Reduce downstream contract defects that are really serialization defects.
5. Keep defect visibility when canonicalization is not safe.

Done looks like
Workflow invocation serialization uses exactly one supported canonical form in final XAML.
Per-argument pseudo-properties are no longer emitted onto invoke activities as if they were final activity attributes.
Argument blob plus per-argument pseudo-property conflicts are eliminated.
Residual JSON-like expression payloads in expression-bearing properties are either:
- deterministically normalized into valid final expression/literal forms
- or preserved as structured defects for downgrade

The dominant defect class shifts away from `unknown_target_argument` caused by malformed invoke serialization.
Contract validation now sees cleaner invoke shapes and reports fewer false positives.
Executable-path defects for leaked JSON expression payloads decrease materially.
No silent fabrication of symbols, arguments, or business values occurs.

Build mode is preserved. Package ZIP and DHG still generate.

Key invariants
- Pattern-first, not activity-first.
- Canonicalize generic invocation serialization, not just one workflow or one activity example.
- Use one authoritative serialization shape for workflow invocation bindings.
- Do not emit both an argument dictionary/blob and per-argument pseudo-properties in the same final invoke node.
- Do not silently delete unresolved executable values.
- If canonicalization is not deterministic, preserve the defect and downgrade.
- Do not hardcode legacy status strings.

Out of scope
- Reworking the overall decomposition strategy
- Inventing missing business logic
- Replacing dependency, graph, or contract validators
- Broad catalog/filter alignment changes
- Deep semantic repair of business rules

Tasks

1. Identify canonical invoke-binding shape
Choose and document one supported final serialization shape for workflow invocation argument bindings used by the pipeline.
All workflow invocation emission paths must converge on this one form.
If multiple equivalent internal representations exist upstream, normalize them before final XAML emission.

2. Remove pseudo-property emission for child arguments
Update the serialization path so child workflow arguments are not emitted as free-form pseudo-properties on invoke activities.
If bindings arrive as per-argument property map entries, transform them into the canonical binding shape before final XAML is produced.
Do not leave both:
- an argument blob/dictionary
- and per-argument pseudo-properties
on the same final invoke activity.

3. Add generic invoke serialization conflict detector
Detect and classify these patterns before final packaging:
- argument blob plus per-argument pseudo-properties
- unsupported invoke binding property names
- child argument bindings emitted in non-canonical shape
- duplicated or conflicting representations of the same binding

When found:
- deterministically canonicalize if safe
- otherwise record structured defects under `invalid_invoke_serialization`

4. Expand residual JSON-expression cleanup coverage
Extend the existing JSON-expression remediation so it covers the remaining code paths that still emit JSON-like payloads in executable properties across the workflow set.
This must be generic across expression-bearing properties, including:
- log message expressions
- assign values
- conditions
- throw expressions
- entity/object payload properties
- invoke binding values
- any other executable property that still receives serialized internal object shapes

5. Add a canonicalization/serialization diagnostics artifact
Emit structured records such as:
- `invokeSerializationFixes[]`
- `residualExpressionSerializationDefects[]`

Each record should include:
- file
- workflow
- activityType
- propertyName
- originalValue
- canonicalizedValue or normalizedValue
- defectType
- normalizationApplied
- normalizationType
- severity
- rationale

6. Integrate with final quality reporting
Add these diagnostics to FinalQualityReport and feed any unresolved invalid invoke serialization or residual expression serialization defects into the current degradation / structural-invalidity derivation path.

7. Add regression tests
Create or extend tests to cover:
- invoke activity with argument blob plus per-argument pseudo-properties
- invoke activity with only canonical binding form
- invoke bindings containing JSON-like expression payload values
- generic executable property still containing JSON-like payload
- deterministic canonicalization success case
- ambiguous/non-deterministic case where defect must remain
- no silent deletion of required executable values
- reduction in unknown_target_argument when the root issue is actually bad invoke serialization
