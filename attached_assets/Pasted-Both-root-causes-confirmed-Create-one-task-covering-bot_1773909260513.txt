FIX 1 — Do not reset classifiedIntent 
to "" on message send

In sendMessageDirect at line 930, remove 
or change the setClassifiedIntent("") call.

The correct behaviour: when a user sends 
a new message, keep the previous 
classifiedIntent value until the server 
sends a new intentClassified event. The 
server classification will overwrite it 
when it arrives. This means:
- If the user previously generated a 
  package and sends "regenerate uipath 
  package", classifiedIntent stays as 
  "UIPATH_GEN" from the previous 
  interaction and the correct message 
  shows immediately
- When the new classification arrives 
  it overwrites correctly as before

Do NOT implement client-side keyword 
matching as an alternative — that was 
discussed and rejected as fragile. The 
keep-previous-value approach is simpler 
and correct.

One edge case to handle: on the very 
first message send when classifiedIntent 
has never been set, it will be "" 
already so there is nothing to preserve. 
The stage fallback applies correctly 
in that case.

FIX 2 — Fix regex replacement spacing 
in catalog auto-remediation

In uipath-integration.ts lines 1272-1276, 
the replacement string $1$2> concatenates 
the content before the removed attribute 
with the content after it. If the removed 
attribute is not surrounded by spaces, 
the adjacent attributes get jammed 
together producing malformed XML like 
attr1=""attr2="value".

Fix the replacement to ensure a space 
is always preserved between attributes:

Change the replacement from:
  `$1$2>\n  ${childElement}\n</${fullTag}>`

To:
  `$1 $2>\n  ${childElement}\n</${fullTag}>`

The added space between $1 and $2 ensures 
proper attribute separation regardless 
of the original whitespace. If $2 is 
empty (the removed attribute was the 
last one before />), the extra space 
before > is harmless in XML.

Also verify the non-self-closing variant 
of this regex if one exists — apply the 
same spacing fix there.

FIX 3 — Add XAML content logging on 
wellformedness failure

When the pre-package XML wellformedness 
check fails, log the full content of the 
failing XAML file to the server logs at 
DEBUG level. This is essential for 
diagnosing future XML errors since the 
content is only held in memory and lost 
when the process fails.

  log.debug(
    `[XAML Wellformedness Failure] 
    ${filename} content:\n${content}`
  )

Log this immediately before or after the 
error is added to the violations list. 
Gate it behind a DEBUG log level so it 
does not appear in production logs by 
default but can be enabled when needed.
