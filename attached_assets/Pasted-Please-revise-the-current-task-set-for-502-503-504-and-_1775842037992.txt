Task #502 correction
--------------------------------------------------
The current version is too aggressive in using catalog defaults / validValues / clrType to resolve all required properties.

Revise #502 so it explicitly introduces a required-property classification model. Every repeatedly failing required property must be classified into exactly one of these buckets:
1. safe-fallback
2. generator-owned
3. spec-required
4. unsupported

Rules:
- Only safe-fallback properties may use catalog-driven fallback resolution.
- Generator-owned properties must be emitted correctly by the generator path, not silently synthesized by the enforcer.
- Spec-required properties must fail early with precise diagnostics if absent; they must not be auto-invented from generic defaults.
- Unsupported properties must fail clearly.

Important examples:
- QueryEntity.EntityType -> spec-required unless proven otherwise
- UpdateEntity.EntityType -> spec-required unless proven otherwise
- CreateEntityRecord.EntityType -> spec-required unless proven otherwise
- CreateFormTask.FormSchemaPath -> spec-required unless proven otherwise
- CreateFormTask.Title / TaskTitle -> do not assume safe fallback unless a deterministic and semantically valid derivation is proven
- ReadTextFile.File -> likely generator-owned or unsupported, not generic-fallback
- SendSmtpMailMessage.Body -> plausible safe-fallback candidate
- While.Condition -> do not solve with generic defaults; treat as expression-sensitive

Revise #502 so that:
- catalog metadata drives detection for all required properties
- catalog-driven fallback is only applied to properties explicitly classified as safe-fallback
- FALLBACK_ELIGIBLE_POLICIES becomes an override / stricter-rules layer on top of the classification model
- the task includes an explicit classification pass for the repeatedly failing properties before any fallback implementation

Also require:
- upstream impact analysis
- downstream impact analysis
- regression risks
- exact files
- explicit out-of-scope

Task #504 correction
--------------------------------------------------
This task is the highest-value structural fix and should be prioritized first.

Keep the core direction:
- spec decomposer output must become a first-class workflow wiring contract
- package assembler must consume executionOrder, invokes, and sharedArguments during initial assembly

- if a required argument cannot be wired safely, fail early with a precise diagnostic rather than generating an empty or semantically useless InvokeWorkflowFile block
- filename and case normalization must be deterministic and validated

Also require:
- graph validation before final package emission
- explicit treatment of executable vs helper workflows
- exact upstream/downstream impact and regression analysis
