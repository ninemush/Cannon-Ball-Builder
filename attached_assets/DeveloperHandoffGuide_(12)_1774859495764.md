### PDD Summary

## 1. Executive Summary
The “birthday greetings V8” project will automate the daily task of checking a dedicated Google Calendar (“Birthdays”) at 8:00 AM and sending personalized birthday greeting emails from the user’s Gmail account. The current process is manual and is sometimes missed due to forgetfulness. The automated solution will run fully unattended on UiPath (no user interaction), read today’s birthday events from the “Birthdays” calendar, resolve each person’s email address from Google Contacts (preferring “Personal” over “Home” when multiple addresses exist), generate a warm, funny, and lightly sarcastic message in the user’s voice using UiPath native Generative AI capabilities (UiPath GenAI Activities), and send the email through the Gmail connector in UiPath Integration Service using the connection **“ninemush@gmail.com”**. If a contact has no email address, the automation will silently skip the person (no email is sent, per SME decision “B”). If message generation fails quality/safety checks, the automation will skip sending for that person and record the reason.

## 2. Process Scope
This automation covers the end-to-end workflow of daily birthday email greetings for events stored in the Google Calendar named **“Birthdays”**. The automation scope begins with a scheduled 8:00 AM trigger and ends when all birthday events for the current day have been processed and the send/skip outcomes are recorded.

Included in scope are: reading birthday events for the day from the “Birthdays” calendar; extracting the full name for each birthday person from the calendar event; looking up the corresponding contact in Google Contacts; selecting the correct email address when multiple addresses exist (with preference for “Personal” over “Home”); generating a personalized email message in the user’s warm/funny/sarcastic voice using UiPath native GenAI; sending the email using the **Gmail Integration Service connector** with connection **“ninemush@gmail.com”**; and persist...

## 1. Automation Architecture Overview

### 1.1 Chosen automation pattern and rationale
**Selected pattern: Single Unattended REFramework + bounded GenAI step (Hybrid RPA + GenAI, not an Agent-led orchestration).**

- **Why not Attended:** only unattended robots are available and the process must be fully autonomous.
- **Why not Action Center / HITL:** the PDD explicitly disallows human-in-the-loop steps in V8 (no approvals/review). Action Center will be used only for **operational escalation** if the platform team chooses to enable it later; it is not in the critical happy path.
- **Why not Maestro:** the workflow is linear (daily schedule → list birthdays → loop → send/skip). BPMN/case management adds overhead without benefit in V8.
- **Why not queue-driven dispatcher/performer:** daily volume is typically low and time-bounded. A single job is sufficient and simpler. If volume grows (e.g., hundreds), we can add Orchestrator Queue fan-out in V9.
- **Why not Communications Mining / Document Understanding:** no inbound email triage or document extraction is in scope.
- **Why GenAI but not Agents:** message generation is a single constrained step; embedding it directly via **UiPath GenAI Activities** inside REFramework provides better determinism and easier operability than introducing Agents (which have inferred availability on this tenant). We’ll still document an optional Agent architecture in Section 8 for future evolution.
