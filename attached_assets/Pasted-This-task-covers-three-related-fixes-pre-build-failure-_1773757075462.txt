This task covers three related fixes: pre-build failure handling, silent error 
swallowing, and redundant database round-trips. Complete them together as they 
share the package state concept introduced here.

PART 1 — Pre-build failure handling
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PROBLEM
When the pre-build fails (document-routes.ts lines 1080–1097), the pipeline sends 
a "skipped" progress message and returns { done: true, package: packageJson } — a 
success response with no .nupkg on disk. The user gets an active download button 
that will either cold-rebuild silently or fail outright.

SOLUTION

1. Introduce a PackageStatus enum in server/types/uipath-package.ts (which already 
exists from Task A):

   export enum PackageStatus {
     BUILDING = 'building',
     READY = 'ready',
     READY_WITH_WARNINGS = 'ready_with_warnings',
     FAILED = 'failed'
   }

2. When pre-build fails, do NOT return { done: true }. Return a structured failure:
   { done: false, status: PackageStatus.FAILED, stage: 'pre_build', 
     error: <error message>, ideaId, timestamp }

3. The progress/SSE message sent to the client must clearly state failure, not 
   "skipped". Example: "Package build failed at pre-build stage — [reason]."

4. Add a file existence check (fs.existsSync or equivalent) before setting package 
   status to READY. Only set READY when the .nupkg is confirmed on disk. Store this 
   as a status field on the package record.

5. Add a retryBuild action that re-triggers the full pipeline for the idea. This 
   should be surfaced in the UI whenever status is FAILED.

6. Download and deploy buttons must only be enabled when status is READY or 
   READY_WITH_WARNINGS.

PART 2 — Silent error swallowing
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PROBLEM
Two locations silently catch and discard errors:
- uipath-pipeline.ts line 223: empty catch {} on AI Center skills fetch
- uipath-pipeline.ts lines 105–106: silent null return on artifact extraction failure

Both produce packages with missing or broken content and give no signal to the user 
or the logs.

SOLUTION

1. Add a warnings array to the pipeline result object:
   type PipelineWarning = {
     code: string
     message: string
     stage: string
     recoverable: boolean
   }

2. Replace the silent catch on AI Center skills fetch with:
   - Structured log: log.warn('AI Center skill fetch failed', { ideaId, 
     error: err.message, timestamp: Date.now() })
   - Push to warnings: { code: 'AI_CENTER_SKILLS_UNAVAILABLE', message: 'AI Center 
     skill references could not be resolved. Affected activities may need manual 
     configuration in Studio.', stage: 'skill_resolution', recoverable: true }

3. Replace the silent null return on artifact extraction failure with:
   - Structured log: log.warn('Artifact extraction failed', { ideaId, 
     error: err.message, timestamp: Date.now() })
   - Push to warnings: { code: 'ARTIFACT_EXTRACTION_FAILED', message: 'One or more 
     artifacts could not be extracted. The package may be incomplete.', 
     stage: 'artifact_extraction', recoverable: false }

4. When the pipeline completes with warnings but the .nupkg was produced, set status 
   to PackageStatus.READY_WITH_WARNINGS (from Part 1).

5. In the frontend package card, when status is READY_WITH_WARNINGS:
   - Show an amber "Review before opening" badge
   - Show warning count inline, e.g. "2 warnings"
   - Make the badge expandable to show each warning message
   - Keep download and deploy buttons enabled

PART 3 — Redundant database round-trips
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PROBLEM
The route handler in document-routes.ts (lines 988–998) loads idea, PDD, SDD, and 
process nodes before calling generateUiPathPackage. The pipeline then calls 
loadIdeaContext internally at line 205, loading the same records again. 
Approximately 4 redundant database queries per generation request.

The IdeaContext interface already exists at line 47 of uipath-pipeline.ts.

SOLUTION

1. Add an optional parameter to generateUiPathPackage:
   generateUiPathPackage(ideaId: string, pkg: UiPathPackage, 
     options?: { preloadedContext?: IdeaContext })

2. If preloadedContext is provided, skip the internal loadIdeaContext call entirely 
   and use the provided data.

3. In document-routes.ts, construct IdeaContext from the data already loaded in the 
   route handler and pass it through options.preloadedContext.

4. Keep loadIdeaContext as a fallback for callers that don't have pre-loaded data 
   (background jobs, retry handlers). Do not remove it.
