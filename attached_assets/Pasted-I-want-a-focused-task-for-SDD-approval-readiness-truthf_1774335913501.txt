I want a focused task for SDD approval/readiness truthfulness.

Context
We now have a concrete bad state in the UI:

- the system posts a warning that the SDD’s deployment artifacts are missing or invalid and that the document cannot be approved until valid artifacts are present
- but the same screen still shows an active “Approve SDD” button
- and also shows a bottom CTA like “SDD is ready for review”

These states are contradictory.

This means that even when `artifactsValid` is false (or the SDD is otherwise not package-ready), the frontend and/or approval rendering path is still treating the document as approvable.

I want this fixed as a correctness/truthfulness issue.

Goal
An SDD that is missing or has invalid deployment artifacts must never appear approvable or “ready for review.”

The readiness truth must be consistent across:
- warning banners/messages
- DocumentCard actions
- bottom CTA / stage action
- server-side approval route enforcement

What I want investigated and fixed

1. Single readiness truth source
Identify and enforce the authoritative readiness source for SDD approval.
This should determine whether the SDD is:
- ready for approval
- blocked
- degraded but approvable
- or invalid

I want the client and server to derive approval state from the same truth model.

2. Frontend rendering consistency
Fix the frontend so that when the SDD is blocked due to invalid/missing artifacts:
- the DocumentCard does not show an active Approve button
- the bottom CTA does not say “ready for review”
- the blocked reason is shown clearly and consistently
- Request Revision remains available if that is the intended recovery path

3. Backend enforcement
Ensure the server-side approval route still blocks approval for invalid artifact state even if the UI somehow drifts.

4. State refresh / stale data

5. Consistent blocked-state UX
Define the expected blocked-state UX for SDD when artifacts are invalid:
- what banner/message appears
- what actions remain available
- what actions are disabled/hidden
- what the bottom status/CTA should say
