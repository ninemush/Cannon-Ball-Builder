Your instinct is right. Here's what's actually happening and why these fixes keep regressing:

The core problem: too many transformation passes competing over the same content.

A single XAML file in this pipeline passes through roughly 10+ sequential transformation stages:

LLM generates the workflow spec
Tree assembler (assembleWorkflowFromSpec) builds raw XAML with buildExpression/ValueIntent
compliancePass runs makeUiPathCompliant — rewrites expressions, types, namespaces
Catalog property protection compares pre/post compliance snapshots
Activity policy filtering removes blocked activities
analyzeAndFix (the analyzer) does another round of auto-fixes
Compliance idempotency loop — runs makeUiPathCompliant AGAIN up to 3 more times
repairMissingImplementation — patches missing implementation blocks
fixMixedLiteralExpressionSyntax — fixes mixed literal/expression patterns
removeDuplicateAttributes — deduplicates XML attributes
injectMissingNamespaceDeclarations — adds missing xmlns
Pre-package validation gate
XML well-formedness gate with targeted fixes
Quality gate with type compatibility repairs
Here's why fixes regress:

When you fixed bare string quoting (e.g., task #282 fixed double-encoding in escapeXml), that fix worked correctly at step 2. But then makeUiPathCompliant at step 3 or 7 would re-process the same expressions and potentially re-introduce the issue, or strip the fix's output because it didn't expect that format. The compliance pass at step 7 is literally re-running the same transformation on already-transformed output, and the log confirms it's often NOT idempotent (the convergence loop runs and sometimes doesn't converge).

The same dynamic explains each of your current 8 errors:

"Asia/Dubai" unquoted — The LLM or tree assembler might emit this correctly as "Asia/Dubai", but a downstream pass sees the / and misinterprets it, or a compliance rewrite strips the quotes because it thinks it's already in brackets.
CStr() LINQ expressions — These are complex enough that the VB lint pass and the compliance pass have different opinions about how they should look. One fixes it, the other "corrects" the fix.
System.Object types — The tree assembler might emit the right type, but the compliance pass's variable block rewriting defaults unfamiliar types back to Object.
The unintended consequence of adding more fix rules:

Every time you add a new expression-fixing rule to any one of these 10+ stages, you risk:

The new rule conflicts with an existing rule in a different stage — because they both try to transform the same expression differently
The compliance idempotency loop amplifies the conflict — it re-runs the transformation, and if the new rule's output doesn't survive re-transformation, it oscillates
The new rule works for the pattern you tested but breaks a pattern another rule was handling — because they share regex/string-matching territory
What this means for the proposed fix approach:

If you just add 3 more expression-fixing rules to the VB lint pass, you'll likely fix the 8 errors you see today, but in 2-3 builds you'll see 4-6 new errors where the new rules interfere with existing compliance or analyzer passes. This is the pattern you've been living.

The systemic fix would be one of two approaches:

Option A (Incremental, lower risk): Instead of adding rules to intermediate passes, add the fixes as late as possible — in a single, final "expression normalization" pass that runs AFTER the compliance idempotency loop converges and BEFORE the quality gate. This pass would have authority to make final corrections without anything downstream re-processing its output. It would need to be explicitly exempt from re-compliance.

Option B (Structural, higher effort): Establish a rule that each expression is transformed exactly once, by exactly one stage. Right now the tree assembler, compliance pass, VB lint, and analyzer all independently try to fix expressions. This would mean defining clear ownership: the tree assembler owns expression generation, the compliance pass owns namespace/structure but does NOT touch expressions, the VB lint owns expression syntax but does NOT touch types, etc. This prevents the "fix-and-re-break" cycle at its root.
