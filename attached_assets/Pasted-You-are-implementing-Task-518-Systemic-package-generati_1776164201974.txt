You are implementing **Task #518 - Systemic package generation quality fix** in CannonBall.

Implement this as a **systemic pipeline refinement**, not a one-off fix for the latest failing run. The goal is to reduce structural invalidity, stubs, and handoff-only outcomes across the package-generation flow.

## Primary objective

Fix these three systemic gaps:

1. **CLI validation is currently a dead end**

   * UiPath CLI validation runs after iterative LLM correction, so CLI defects are logged but never acted on.
   * `cliDefectsToHealingInput()` exists in `server/uipath-cli-validator.ts` but is not wired into the correction flow.
   * The corrector currently only consumes internal quality-gate violations.

2. **Emission approval is bottlenecked on namespace lookup**

   * `catalog/run-emission-triage.js` still checks namespace mappings by scraping hardcoded entries from `xaml-compliance.ts`.
   * The catalog service already exposes DLL-derived namespace data via `getAllPackageNamespaceEntries()`.
   * There are about **227 activities** with `propertiesComplete: true` and `emissionApproved: false` that are safe approval candidates blocked only by this namespace lookup gap.

3. **The AI still generates XAML too blindly**

   * The enrichment prompt gives schema and filtered catalog context, but no concrete XAML structural patterns.
   * We already have completed project XAML and Studio baselines. Use them to teach structural patterns, not business logic.

## Non-negotiable design constraints

### A. Do NOT run the iterative corrector twice

Do **not** implement a second iterative LLM correction pass after CLI.

Why:

* the corrector verifies fixes using internal quality-gate revalidation
* CLI defect resolution is not reliably visible to the internal QG
* a second pass would likely trigger false “no improvement” reverts
* it would also add cost and increase oscillation risk

Correct design:

1. internal/meta validation
2. **pre-correction CLI analyze-only**
3. convert and normalize CLI defects
4. merge them into the existing correction input
5. run the existing iterative corrector **once**
6. rebuild
7. final CLI validation (`analyze + pack`) as the verification gate

### B. Use the correct CLI functions

* Use **`runCliAnalyze()`** for the pre-correction pass
* Use **`runCliValidation()`** only for the final post-correction verification pass
* Do **not** run `analyze + pack` twice

### C. CLI defects must be normalized to the corrector’s model

A naive merge into `qgViolationsForIterative` will silently fail.

Current mismatch:

* CLI healing input is CLI-shaped, e.g. `{ source, ruleId, severity, file, line?, message }`
* the corrector expects `QualityGateIssue`-like objects, e.g. `{ check, file, detail, severity }`
* the corrector filters via `FIXABLE_QG_CHECKS.has(issue.check)`
* the corrector verifies improvement via internal `runQualityGate()`, which cannot observe most CLI rule resolution

Therefore, you must implement all of the following:

1. **Normalize CLI defects into corrector-compatible shape**

   * map `ruleId -> check`
   * map `message -> detail`
   * preserve file / line / severity / source where useful
   * do not pass raw CLI defect objects directly into `qgViolationsForIterative`

2. **Make CLI-derived issues survive fixable-check filtering**
   Choose one clean approach:

   * map known CLI rule IDs to existing internal fixable check names where there is a strong semantic match, or
   * introduce a small controlled set of CLI-specific check names and register them in `FIXABLE_QG_CHECKS`

   Do not leave CLI issues as opaque `ST-*` checks if that means they will all be dropped.

3. **Do not let CLI-driven fixes be reverted solely because internal QG cannot verify them**

   * preserve existing revert safety for internal QG issues
   * for CLI-sourced issues, use **deferred verification**
   * the final `runCliValidation()` pass is the authoritative verifier for CLI-driven fixes
   * do not globally weaken revert logic; scope this carefully

4. **If a CLI rule has no safe mapping or fix strategy**

   * keep it visible in diagnostics
   * do not force it through the iterative corrector unless a safe explicit handling path exists

### D. Keep emission expansion safe

Only approve candidates that satisfy all of:

* `propertiesComplete: true`
* package namespace metadata exists from DLL extraction
* package/activity satisfies the rest of triage safety conditions

Do **not** approve incomplete candidates.

### E. Keep XAML prompt additions compact

The enrichment stage has a 45-second timeout and substantial existing context.
Add only **2-3 focused structural snippets**, under roughly **1000 tokens total**.
Do not paste full workflows.
Do not include presentation-only noise like:

* `sap:HintSize`
* `sap2010:WorkflowViewState`

### F. Watch total pipeline duration

Adding pre-correction CLI analyze plus final CLI validation will increase total runtime.

Requirements:

* measure before/after pipeline duration
* avoid redundant pack/build steps
* improve correctness without disproportionate runtime growth
