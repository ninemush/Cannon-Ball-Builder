#464 - Benchmark-Hit Required Property Source Binding and Safe Fallback Authority

What & Why
Recent benchmark runs now show a much narrower dominant failure class than earlier phases. The package build succeeds, but the final output is still structurally invalid because certain benchmark-hit required properties repeatedly fail with the same pattern:

- no valid source binding found
- generic default or empty placeholder rejected
- no contract-valid fallback emitted
- activity left structurally incomplete

The repeated benchmark-hit cases now include at least:
- LogMessage.Message
- If.Condition
- QueryEntity.EntityType

The current problem is no longer mainly XML corruption, typed-property-object leakage, or invoke contract emptiness. The current dominant problem is that the pipeline still cannot reliably produce or preserve valid required-property values for benchmark-hit activities, and when authoritative business values are absent, it has no safe fallback policy that keeps the XAML structurally valid.

This task must not be a broad schema redesign. It must narrowly fix source-binding and safe fallback authority for the benchmark-hit activity/property pairs proven by current evidence.

Done looks like
- Benchmark-hit required-property failures for the targeted activity/property pairs drop materially on the Birthday benchmark.
- LogMessage activities no longer reach validation without Message when a valid upstream source exists or a safe fallback is allowed.
- If activities no longer reach validation with blocked generic-default conditions when a valid source or safe fallback is available.
- QueryEntity activities no longer reach validation without EntityType when a valid source exists or a safe placeholder policy applies.
- The pipeline distinguishes clearly between:
  - valid bound value
  - safe structural fallback
  - hard block because no valid value and no safe fallback policy exists
- Existing validators remain strict, but they are no longer the first place these benchmark-hit properties are discovered as missing.

Out of scope
- Broad required-property redesign across all activities
- Graph connectivity fixes
- Workflow contract/interface redesign
- Typed-property-object boundary work
- Prompt engineering
- Weakening quality gate or final validation
- Inventing business logic where no safe fallback exists
- A family-wide rewrite of all templates

Design rules
1. Benchmark-hit only
This task is limited to activity/property pairs confirmed by benchmark evidence. Do not broaden to every required property in the system.

2. Source binding first
If a valid upstream source exists, prefer binding that source rather than inventing a fallback.

3. Safe fallback second
If no authoritative source exists, only emit a fallback when it is both:
- structurally valid for the activity/property contract
- safe from a business-logic standpoint
If those conditions are not met, emit a clear diagnostic/block instead of a guessed value.

4. No generic invalid defaults
Do not use generic defaults like empty string or False when the downstream validator or activity contract already rejects them as invalid substitutions for the targeted benchmark-hit properties.

5. Preserve strict validation
Do not weaken validators. Fix the value selection and fallback policy upstream.

Tasks
1. Confirm the benchmark-hit failing activity/property pairs from current evidence
Using the latest Birthday benchmark artifacts and validation output, confirm the exact repeated failing pairs to target in this task.
Start with:
- LogMessage.Message
- If.Condition
- QueryEntity.EntityType
Only include additional pairs if the current benchmark evidence clearly shows the same source-binding/fallback failure pattern.

2. Trace the value-selection path for each targeted pair
For each confirmed activity/property pair, trace:
- where the property value is first sourced
- where it becomes empty, generic, or invalid
- whether a valid upstream source was available but missed
- where the required-property-enforcer accepts, rejects, or blocks the value
- whether a safe structural fallback is already partially implemented but not applied

3. Fix source-binding for benchmark-hit properties
For each targeted pair, ensure the pipeline correctly selects a valid upstream source when one exists.
This fix should be narrow and evidence-backed. Do not generalize beyond the benchmark-hit properties unless the same code path is clearly shared.

4. Add safe fallback authority for benchmark-hit properties
For each targeted pair, define and implement a safe fallback policy only if one is justified.
Examples of what this means:
- LogMessage.Message may allow a structurally valid safe placeholder message when no authoritative message exists
- If.Condition may allow a safe structural fallback only if it does not silently change business behavior in an unsafe way
- QueryEntity.EntityType may allow a safe placeholder only if the target activity contract and benchmark logic make that acceptable
If no safe fallback exists, keep the block and record a clear diagnostic.

5. Prevent invalid generic defaults from being emitted for the targeted pairs
Stop the pipeline from emitting values already known to be rejected for these benchmark-hit properties, such as:
- empty Message
- blocked generic False condition
- empty or invalid EntityType
for the confirmed targeted paths.

6. Add lightweight diagnostics
Extend an existing reporting surface so benchmark/debug runs can show for each targeted pair:
- activity type
- property name
- upstream source found or not
- value chosen
- fallback applied or not
- reason for block if still unresolved
Keep this lightweight. No new diagnostics framework.

7. Add focused regression tests
Add tests covering:
- LogMessage.Message binds valid upstream source when present
- LogMessage.Message uses only an allowed safe fallback when source is absent
- If.Condition no longer emits a blocked generic default on the benchmark-hit path
- QueryEntity.EntityType binds valid source or uses only an allowed safe fallback
- unresolved targeted pairs produce explicit diagnostics rather than structurally incomplete emission
- benchmark-style repeated missing-property failures are materially reduced
