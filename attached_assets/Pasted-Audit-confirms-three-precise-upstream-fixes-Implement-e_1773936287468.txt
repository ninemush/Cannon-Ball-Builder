UPSTREAM FIX 1 — Fix looksLikeVariableRef 
bypass at all six write points

The six write points (B, C, E, G, N, Q) 
gate bracketing behind looksLikeVariableRef 
which rejects any expression containing 
operators, spaces, or parentheses — 
exactly the expressions that need 
brackets most.

Change the logic at all six write points 
from:

  if (looksLikeVariableRef(value)) {
    // bracket wrap
  } else {
    // use raw value
  }

To:

  // Always bracket-wrap unless the value 
  // is already bracketed, a quoted string 
  // literal, or a known safe literal
  if (!isAlreadyBracketed(value) && 
      !isQuotedStringLiteral(value) && 
      !isSafeLiteral(value)) {
    value = `[${value}]`
  }

Where:
  isAlreadyBracketed: value starts with 
    [ and ends with ]
  isQuotedStringLiteral: value matches 
    ^".*"$ or ^&quot;.*&quot;$
  isSafeLiteral: value is exactly True, 
    False, Nothing, null, or empty string

This means int_SeverityCode <> 10 gets 
bracketed to [int_SeverityCode <> 10]. 
Inside brackets, UiPath's expression 
evaluator handles VB syntax. The XML 
parser never sees the raw <>.

Do NOT apply escapeXml to VB expressions 
that should be bracketed — escaping 
<> to &lt;&gt; produces valid XML but 
wrong VB semantics. Bracketing is the 
correct fix, not escaping.

Apply this change at write points B, C, 
E, G (xaml-generator.ts) and N, Q 
(workflow-tree-assembler.ts).

Write point H (line 2074) already calls 
escapeXml after — this is wrong for VB 
expressions. Change H to use the same 
bracket-first logic above, removing 
the escapeXml call for the Value 
argument specifically. The To argument 
(OutArgument) is already handled 
correctly with ensureBracketWrapped.
