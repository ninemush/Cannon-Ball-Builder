## CONTEXT

You are working on an existing application called **CannonBall** — an AI-first automation
platform that lets Process SMEs generate and deploy UiPath automations through a
conversational interface. The platform already pushes queue items to UiPath Orchestrator.

This prompt rebuilds and hardens the entire UiPath integration layer from scratch. The
current integration is fragile and fails on every push. That is not acceptable.

**Core Principles — non-negotiable:**
1. AI/automation first. Every integration action is initiated and managed by AI.
2. Resilience. Failures are handled gracefully, retried intelligently, and surfaced clearly.
3. Smart provisioning. Robots are provisioned only when needed and released when done.
4. UX/UI quality matches the integration quality. Users should never see a raw error or a
   spinner with no context.

## PART 2 — UIPATH INTEGRATION LAYER REBUILD

### 2.1 — Authentication Module (`uipath_auth.py`)

Build a singleton authentication manager. This is the foundation everything else depends on.

**Requirements:**

- Uses Client Credentials OAuth2 flow (not browser-based)
- Reads all credentials from environment variables:
  ```
  UIPATH_CLIENT_ID
  UIPATH_CLIENT_SECRET
  UIPATH_BASE_URL          # e.g. https://cloud.uipath.com/org/tenant/orchestrator_
  UIPATH_TENANT_NAME
  UIPATH_ORGANIZATION_ID
  UIPATH_FOLDER_ID         # numeric folder ID for target folder
  ```
- Token endpoint: `https://cloud.uipath.com/identity_/connect/token`
- Scopes: `OR.Execution OR.Queues OR.Processes OR.Folders.Read OR.Jobs OR.Triggers`
- Caches the token in memory with its expiry timestamp
- `get_token()` method: returns valid token, refreshes automatically if within 60 seconds
  of expiry — never lets a call fail due to an expired token
- `get_headers()` method: returns the full header dict for any Orchestrator API call:
  ```python
  {
    "Authorization": f"Bearer {token}",
    "X-UIPATH-OrganizationUnitId": UIPATH_FOLDER_ID,
    "Content-Type": "application/json"
  }
  ```
- `health_check()` method: calls `GET /odata/Folders` and returns `(bool, str)` — True if
  reachable, False with an error message if not
- All token failures log a structured error and raise a `UiPathAuthError` custom exception
- Never log or expose the client secret. Log only the first 6 characters of the client ID
  for traceability.

### 2.2 — Orchestrator API Client (`orchestrator_client.py`)

A clean, typed wrapper around every Orchestrator API endpoint CannonBall needs. No raw
`requests` calls anywhere else in the codebase — all HTTP goes through this client.

**Design:**
- Takes an instance of the auth manager in its constructor
- Every method auto-injects headers via `auth.get_headers()`
- Every method has a `max_retries=3` parameter with exponential backoff (1s, 2s, 4s)
- All responses are checked for status codes. Non-2xx raises a `OrchestratorAPIError`
  with the status code, endpoint, and response body included in the error message
- Methods return typed Python dicts or dataclasses — never raw response objects

**Required methods:**

### 2.3 — Smart Provisioning Module (`smart_provisioning.py`)

This is the intelligence layer. It behaves like infrastructure auto-scaling but for UiPath
robots. It decides when to spin work up and when to release it.

**Design Philosophy:**
- No robot runs unless there is a reason to run it
- Provisioning decisions are made by AI, not static rules
- The system learns from queue behavior over time (within a session at minimum)
- All provisioning decisions are logged with reasoning

**Class: SmartProvisioningEngine**

```python
class SmartProvisioningEngine:
    def __init__(self, orchestrator: OrchestratorClient, config: ProvisioningConfig)
```

**ProvisioningConfig (dataclass):**
```python
@dataclass
class ProvisioningConfig:
    process_name: str
    queue_name: str
    min_robots: int = 0           # 0 = fully elastic, spin down to zero
    max_robots: int = 5           # hard cap per process
    scale_up_threshold: int = 10  # queue items per robot before adding another
    scale_down_threshold: int = 2  # items per robot before releasing one
    cooldown_seconds: int = 60    # min time between scaling decisions
    sla_minutes: int = 60         # target completion time for queue
    idle_timeout_seconds: int = 300  # release robot after this many seconds idle
```

**Methods:**

```python
def analyze_queue_and_decide(self, queue_stats: dict, active_jobs: list) -> ProvisioningDecision
```
- Calls Claude API with queue stats, active job count, SLA, and historical pattern
- System prompt: "You are a provisioning intelligence engine for UiPath robot fleets.
  Analyze the queue metrics and active jobs below. Decide whether to scale up, scale down,
  or hold. Return JSON only: {action: 'scale_up'|'scale_down'|'hold', robots_delta: int,
  reasoning: str, confidence: float}. Consider: queue depth trend, average processing time,
  SLA target, and cooldown period."
- Returns a `ProvisioningDecision` dataclass with the action, delta, and reasoning

```python
def execute_decision(self, decision: ProvisioningDecision) -> ExecutionResult
```
- If scale_up: calls `orchestrator.start_job()` for `decision.robots_delta` jobs
- If scale_down: identifies idle jobs and calls `orchestrator.stop_job()` gracefully
- If hold: logs the hold decision with reasoning, no API calls
- Returns execution result with job IDs started or stopped

```python
def monitor_loop(self, interval_seconds: int = 30) -> None
```
- Runs in a background thread
- Every `interval_seconds`, fetches queue stats and active jobs
- Calls `analyze_queue_and_decide`
- If cooldown has elapsed and action is not hold, calls `execute_decision`
- Logs every cycle to the provisioning_log table

```python
def get_provisioning_status(self) -> dict
```
- Returns current state: active jobs, queue depth, last decision, next check timestamp

```python
def emergency_stop(self, process_name: str) -> dict
```
- Stops all running jobs for the given process immediately
- Disables any active queue triggers
- Returns count of jobs stopped

### 2.6 — Agent Generation Pipeline (Updated with Resilience)

Replace the existing `full_pipeline()` with this hardened version.

**Key changes from the original spec:**
- Test execution is now a blocking gate before publish (Stage 7.5)
- Smart provisioning is configured at deploy time (Stage 9.5)
- Action Center monitoring starts automatically after deploy
- Every stage writes status to the database, not just memory
- Failed stages can be retried individually from the UI without restarting the full pipeline
- Max 2 retries on LLM generation stages with different seeds if code quality check fails

**Updated 12-Stage Pipeline:**

```
Stage 1  → SME describes agent in natural language
Stage 2  → Intent extraction + agent type validation (agent vs. robot vs. hybrid)
Stage 3  → Template selection from CannonBall library
Stage 4  → main.py generation via LLM
Stage 5  → Code quality gate: check for placeholder comments, syntax errors, missing types
           → If fails: retry up to 2x with corrected prompt. If still fails: surface to SME.
Stage 6  → Supporting file generation (pyproject.toml, langgraph.json, uipath.json)
Stage 7  → Generate test cases via AI
Stage 7.5→ uipath init + uipath pack (package only, do not publish yet)
Stage 8  → Run test execution against packed agent
           → Evaluate results with AI
           → BLOCKING: if AI evaluates as 'fail', pipeline stops. Surfaces report to SME.
Stage 9  → uipath auth (headless) + uipath publish to Tenant Process Feed
Stage 9.5→ Smart Provisioning configuration: AI suggests initial thresholds based on
           intent complexity. Creates queue trigger. Starts monitor loop.
Stage 10 → Validation: poll Orchestrator to confirm process exists and is active
Stage 11 → Action Center monitoring activated for this process
Stage 12 → Handoff report delivered to SME in CannonBall UI
```

## PART 5 — ENVIRONMENT SETUP AND RESILIENCE RULES

### 5.1 — Required Environment Variables

All must be in Replit Secrets. Add validation on startup: if any required variable is
missing, log a clear error and refuse to start rather than failing silently mid-operation.

```
# UiPath Core

UIPATH_CLIENT_SECRET
UIPATH_BASE_URL
UIPATH_TENANT_NAME
UIPATH_ORGANIZATION_ID
UIPATH_FOLDER_ID

# Anthropic
ANTHROPIC_API_KEY

# Database
DATABASE_URL

# App
APP_SECRET_KEY
ENVIRONMENT   # development | production
```

### 5.2 — Resilience Rules

Every rule below is a hard requirement, not a recommendation.

1. **No silent failures.** Every exception must be caught, logged with context, and
   either retried or surfaced to the UI. `except: pass` is banned.

2. **No hardcoded values.** All credentials, URLs, thresholds, and configuration are
   environment variables or database-stored config. No exceptions.

3. **No in-memory state for anything that matters.** Pipeline job status, provisioning
   config, action tasks, and test results are all written to the database immediately.
   A server restart should not lose any user data or job state.

4. **All subprocess calls must:**
   - Use list format, not shell strings
   - Have `timeout=120` set
   - Capture stdout and stderr separately
   - Log the command being run (without credentials)
   - Return a structured result, never raise unhandled exceptions

5. **All HTTP calls must:**
   - Use the auth manager's `get_headers()` — never build auth headers manually
   - Have retry logic with exponential backoff (max 3 retries)
   - Have a request timeout of 30 seconds
   - Log the endpoint and status code on every response

6. **Token refresh must be proactive.** Refresh 60 seconds before expiry, not after it
   fails. A 401 on an Orchestrator call should be treated as a signal to refresh and retry
   once — not to surface an error to the user.

7. **Background threads must be managed.** The provisioning monitor loop and action
   center polling thread must be registered in an app-level thread registry. On shutdown,
   all threads are stopped cleanly. No orphaned threads.

8. **Database writes are synchronous for status updates.** Do not fire-and-forget status
   writes. If the DB write fails, log it and surface a warning — but do not fail the
   pipeline because of a log write.

9. **The `.nupkg` is validated before publish.** After `uipath pack`, check that the file
   exists, is non-zero bytes, and is less than 100MB. If any check fails, surface a clear

### 7.1 — SDK Environment Setup (`sdk_environment.py`)

The UiPath Python SDK must run in an isolated virtual environment per job, not in the
server's global Python environment. This is non-negotiable -- mixing SDK state across
jobs causes authentication bleed and dependency conflicts.

**Class: SDKEnvironmentManager**

```python
class SDKEnvironmentManager:
    BASE_DIR = "/tmp/cannonball_jobs"

    def create_job_environment(self, job_id: str) -> JobEnvironment
```

Creates the following structure for every job:

```
/tmp/cannonball_jobs/{job_id}/
    venv/                    # isolated Python virtual environment
    project/                 # generated agent files land here
        main.py
        pyproject.toml
        langgraph.json
        uipath.json
    logs/                    # stdout/stderr from every subprocess call
        auth.log
        init.log
        pack.log
        publish.log
    artifacts/               # output files
        *.nupkg
```

```python
def bootstrap_venv(self, job_env: JobEnvironment) -> tuple[bool, str]
```

Creates a fresh virtual environment and installs the base SDK:

```bash
python3 -m venv /tmp/cannonball_jobs/{job_id}/venv
/tmp/cannonball_jobs/{job_id}/venv/bin/pip install --quiet \
    uipath-langchain>=0.5.0 \
    uipath>=1.0.0 \
    langchain-core \
    langgraph
```

All subsequent subprocess calls for this job use the venv Python and pip:
`/tmp/cannonball_jobs/{job_id}/venv/bin/python`
`/tmp/cannonball_jobs/{job_id}/venv/bin/uipath`

Never use the system `uipath` or `python` commands for job-specific operations.

```python
def install_dependencies(self, job_env: JobEnvironment, packages: list[str]) -> tuple[bool, str]
```

Installs additional packages into the job's venv after dependency detection runs.
Runs: `venv/bin/pip install {packages}` with a 120-second timeout.
Returns success/failure and pip's output.

```python
def teardown_environment(self, job_id: str, keep_logs: bool = False) -> None
```

On success: removes the entire job directory after 24 hours via scheduled cleanup.
On failure: removes `venv/` and `project/` but keeps `logs/` and `artifacts/` for debugging.
Never delete on failure without keeping logs -- this is how you debug what went wrong.

```python
def get_venv_python(self, job_id: str) -> str
```

Returns the absolute path to the venv Python binary for this job.
Raises `EnvironmentNotFoundError` if the venv does not exist.

### 7.2 — Dynamic Dependency Detection (`dependency_detector.py`)

The AI generates `main.py`. That code imports libraries. Those libraries must be in
`pyproject.toml` before `uipath pack` runs or the pack step fails. This module bridges
that gap automatically.

**Class: DependencyDetector**

```python
def extract_imports_from_code(self, code: str) -> list[str]
```

Parses the generated `main.py` using Python's `ast` module (not regex).
Extracts all `import X` and `from X import Y` statements.
Returns a list of top-level module names.

Example input:
```python
import requests
from bs4 import BeautifulSoup
from langchain_community.tools import DuckDuckGoSearchRun
import json  # stdlib, ignore
```
Returns: `["requests", "beautifulsoup4", "langchain-community"]`

Note: stdlib modules must be filtered out. Use `sys.stdlib_module_names` (Python 3.10+)
or a maintained list for earlier versions. Do not add stdlib modules to pyproject.toml.

```python
def resolve_package_names(self, import_names: list[str]) -> dict[str, str]
```

Import names do not always match PyPI package names (`bs4` → `beautifulsoup4`,
`PIL` → `Pillow`, `sklearn` → `scikit-learn`). Maintain a mapping dict of known
mismatches. For unknown imports, call Claude API:

System prompt: "You are a Python packaging expert. Given an import name, return the
correct PyPI package name. Return JSON only: {import_name: pypi_package_name}.
If the import is a stdlib module, return null as the value."

Return a dict mapping import names to PyPI package names, with nulls filtered out.

```python
def validate_packages_exist(self, packages: list[str]) -> dict[str, bool]
```

For each package, check PyPI availability via:
`GET https://pypi.org/pypi/{package}/json` with a 5-second timeout.
Returns dict of package name → exists (bool).
Log any packages that don't exist on PyPI -- these are likely hallucinated by the LLM
and need to be flagged to the SME before proceeding.

```python
def build_pyproject_toml(self, intent: dict, detected_packages: list[str]) -> str
```

Generates the final `pyproject.toml` with:
- Base dependencies always included: `uipath-langchain>=0.5.0`, `langgraph`, `langchain-core`
- Detected packages from `extract_imports_from_code` appended dynamically
- Version pins omitted by default (use latest) unless the package is known to have
  breaking changes (maintain a pinned_versions dict for these)

Returns the complete pyproject.toml content as a string.

```python
def detect_and_inject(self, job_env: JobEnvironment, main_py: str, intent: dict) -> DependencyResult
```

Orchestrates the full flow:
1. `extract_imports_from_code(main_py)`
2. `resolve_package_names(imports)`
3. `validate_packages_exist(packages)` -- halt if any package not found on PyPI,
   surface to SME with: "The AI generated code that imports [package], which doesn't
   exist on PyPI. This is likely an error. Please review the generated code."
4. `build_pyproject_toml(intent, packages)` -- write to project directory
5. `install_dependencies(job_env, packages)` -- install into venv
6. Return `DependencyResult` with: packages detected, packages installed, any warnings

### 7.3 — Machine and Robot Prerequisite Checks (`prerequisite_checker.py`)

Before any publish or job start, CannonBall must verify that the Orchestrator environment
can actually execute the agent. Silent failures from missing machines or robots end here.

**Class: PrerequisiteChecker**

```python
def check_all(self, process_type: str = "agent") -> PrerequisiteReport
```

Runs all checks below in sequence and returns a structured report. If any check is
`BLOCKING`, the pipeline halts and surfaces the report to the SME before proceeding.

```python
def check_machine_availability(self) -> CheckResult
```

Calls `orchestrator.get_robots()` for the target folder.
Checks:
- At least one machine is registered in the folder → BLOCKING if zero
- At least one machine is in "Available" or "Busy" state (not "Disconnected") → WARNING if all disconnected
- Machine has sufficient license type for the process (Unattended for background agents) → BLOCKING if wrong type

Returns:
```python
CheckResult(
    name="Machine Availability",
    status="pass"|"warning"|"blocking",
    detail="2 machines registered, 1 available (Unattended)",
    remediation="Register a machine in Orchestrator > Machines before deploying."
)
```

```python
def check_robot_license(self) -> CheckResult
```

Calls Orchestrator license endpoint to verify:
- Unattended robot licenses are available (not exhausted)
- If license count is at 100% utilization: WARNING (not blocking -- a license may free up)
- If license count is zero: BLOCKING

```python
def check_folder_permissions(self) -> CheckResult
```

Verifies the External App credentials have write access to the target folder by
attempting a lightweight write operation (add then immediately delete a test queue item
in a cannonball_health_check queue). If the queue doesn't exist, create it first.
BLOCKING if write fails.

```python
def check_package_feed_writable(self) -> CheckResult
```

Verifies the target package feed accepts uploads by checking if the External App has
the `OR.Execution.Write` scope active. Cross-reference against the scopes returned
during the integration setup (these are already stored in the DB from the admin wizard).
BLOCKING if scope is missing.

```python
def check_action_center_license(self) -> CheckResult
```

Calls `orchestrator.get_action_catalog()`. If it returns a 403 or 404, Action Center
is not licensed or not enabled.
Status: WARNING (not blocking -- the pipeline can proceed, but Action Center monitoring
will be disabled for this process and the SME is notified).

```python
def check_test_manager_license(self) -> CheckResult
```

Calls the Test Manager endpoint. If unreachable or unlicensed:
Status: WARNING with message: "Test Manager is not available on this tenant.
Test execution (Stage 8) will be skipped. Publish will proceed without test gate."
The pipeline continues but logs that the test gate was bypassed and why.

```python
def generate_prerequisite_report(self, results: list[CheckResult]) -> str
```

Returns a markdown string for display in the CannonBall UI. Format:

```
## Pre-deployment Check

✓  Machine Availability    2 machines registered, 1 available
✓  Robot License           3 of 5 Unattended licenses in use
✓  Folder Permissions      Write access confirmed
✓  Package Feed            Upload access confirmed
⚠  Action Center           Not licensed — exception routing disabled for this process
⚠  Test Manager            Not detected — test gate will be skipped

2 warnings. No blocking issues. Ready to deploy.
[Proceed to Deploy]   [Cancel]
```

If any blocking issues exist:
```
✗  Machine Availability    No machines registered in SumeetAppDefaultFeed

1 blocking issue. Deployment cannot proceed until resolved.
[View Remediation Steps]   [Cancel]
```

### 7.4 — Process Lifecycle Manager (`process_lifecycle.py`)

After publish, a process record must exist in Orchestrator before jobs can run against it.
This module manages the full create → version → deprecate lifecycle.

**Class: ProcessLifecycleManager**

```python
def ensure_process_exists(self, agent_name: str, package_version: str) -> ProcessRecord
```

The most important method. Called after every successful `uipath publish`.

Logic:
1. Call `orchestrator.get_process_by_name(agent_name)`
2. If process does not exist:
   → Call `orchestrator.create_process(agent_name, package_version)`
   → Log: "Created new process: {agent_name} v{package_version}"
3. If process exists but version is older:
   → Call Orchestrator PATCH to update the process to the new package version
   → Log: "Updated process {agent_name} from v{old} to v{package_version}"
4. If process exists and version matches:
   → No action needed, log: "Process {agent_name} v{package_version} already current"
5. Poll `orchestrator.get_process_by_name` every 5 seconds for up to 60 seconds
   to confirm the process is in "Available" state before returning
6. If not Available after 60 seconds: raise `ProcessActivationTimeout`

Returns a `ProcessRecord` dataclass with: process_key, name, version, status, folder_id.

```python
def get_process_versions(self, agent_name: str) -> list[ProcessVersion]
```

Returns all versions of a process that have been published, with timestamps and
deploy status. Used in the CannonBall UI to show version history per agent.

```python
def rollback_process(self, agent_name: str, target_version: str) -> ProcessRecord
```

Updates the Orchestrator process to point to a previous package version.
Useful when a newly deployed agent has runtime errors.
Logs the rollback with who triggered it and why (requires a reason string parameter).

```python
def deprecate_process(self, agent_name: str) -> bool
```

Sets the process to disabled in Orchestrator without deleting it.
Stops any active queue triggers for this process.
Does NOT stop currently running jobs -- those complete naturally.
Returns True if successfully disabled.

```python
def get_process_health(self, agent_name: str) -> ProcessHealthReport
```

Returns a snapshot of the process's current health:
- Status: Available / Disabled / Faulted
- Last job: timestamp and outcome
- Job success rate over last 7 days (if history available)
- Active trigger status
- Queue depth (if queue trigger configured)

The 12-stage pipeline now incorporates all of the above. Replace the stage list in
Part 2.6 with this complete version:

```
Stage 1   SME describes agent in natural language via CannonBall chat interface

Stage 2   Intent extraction
          → extract_intent() returns structured dict
          → Validates: agent_name is unique in Orchestrator (calls get_process_by_name)

          → Quality gate: check for "# TODO", "pass" as function body, missing type hints
          → Max 2 retries with corrected prompt if quality gate fails
          → If both retries fail: surface generated code to SME in inline editor
            with specific issues highlighted. SME can edit and re-trigger from Stage 4.

Stage 5   Dependency detection and injection
          → dependency_detector.detect_and_inject()
          → Extracts imports from generated main.py
          → Resolves PyPI package names
          → Validates all packages exist on PyPI
          → BLOCKING if any package not found: surface to SME with specific package name

          → WARNING issues are logged and noted in handoff report but do not block
          → Test Manager and Action Center availability determined here for Stage 8 and 11

Stage 8   SDK pipeline — package only, do not publish yet
          → Run in job venv: uipath auth (headless, client credentials)
          → Run: uipath init (generates entry-points.json from code structure)
          → Run: uipath pack (produces .nupkg)
          → Validate .nupkg: exists, non-zero bytes, under 100MB

            → generate_test_cases() from intent and main.py
            → create_test_set() in Test Manager
            → run_tests_and_wait() with 300 second timeout
            → evaluate_results() via Claude
            → If AI evaluates as FAIL: halt, surface test report to SME
              SME can: edit code and retry from Stage 4, or override with typed confirmation
          → If Test Manager not licensed:
            → Log bypass, notify SME via UI banner, proceed to Stage 10

Stage 10  Publish
          → uipath publish to Tenant Process Feed (not --my-workspace)
          → Capture process configuration link from stdout
          → Upload confirmation logged to database

Stage 11  Process lifecycle management
          → process_lifecycle.ensure_process_exists()
          → Creates or updates process in target Orchestrator folder
          → Polls until process status is Available (60 second timeout)
          → BLOCKING if process not Available after timeout

Stage 11.5 Smart Provisioning configuration
          → AI suggests initial thresholds based on intent complexity and process type
          → Creates queue trigger in Orchestrator via orchestrator.create_queue_trigger()
          → Starts provisioning monitor loop for this process
          → Logs initial config to provisioning_log table

Stage 12  Handoff
          → Action Center monitoring activated for this process (if licensed)
          → Handoff report generated and delivered to SME in CannonBall UI:
            - Process name and version
            - Orchestrator process link
            - Package feed location
            - Test results summary (or bypass notice)
            - Prerequisite warnings from Stage 7
            - Initial provisioning config
            - Links: Open in Orchestrator, View Provisioning Dashboard, View Test Report
