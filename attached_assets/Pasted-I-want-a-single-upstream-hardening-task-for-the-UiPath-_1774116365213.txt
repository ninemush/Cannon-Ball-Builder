- TO-BE / solution design shaping
- artifact generation decisions
- deployment/provisioning into UiPath

Two things matter here:
1. resilience to UiPath changes (endpoint drift, scope drift, service behavior changes)
2. truthful system/UI behavior (no optimistic “available” or “successful” states when reality is partial, inferred, or failed)

Important:
Before proposing implementation, I want an impact analysis of what this change will affect so we do not fix one layer while breaking another.

What I want from you

Phase 1 — Impact Analysis
Please first analyze and report:

1. Current architecture impact surface
Identify all areas touched by UiPath integration truth:
- capability discovery / probe flows
- TO-BE / PDD / SDD generation inputs
- artifact generation decisions
- deployment/provisioning
- frontend status displays / availability displays / deployment results
- admin/settings connectivity and diagnostics
- package generation decisions if any consume capability truth

2. Dependency graph of affected modules
Map the main modules and their relationships, including at minimum:
- uipath-auth.ts
- uipath-fetch.ts

- uipath-integration.ts
- automation-hub.ts
- artifact provisioner
- relevant routes
- any metadata service / endpoint service
- any UI components/pages showing availability, probe results, or deployment outcomes

3. Truthfulness risk analysis
Identify where the current system may be optimistic or ambiguous, for example:
- service marked available when only inferred
- service marked available on weak/ambiguous probe results
- deployment presented as success when partially failed
- UI flattening configured/reachable/licensed/confident into one status
- duplicated endpoint or auth logic creating inconsistent results between layers

4. Drift resilience analysis
Identify where UiPath changes would currently hurt us:
- hardcoded endpoint roots
- duplicated token logic
- hardcoded scopes
- weak probe semantics
- lack of centralized endpoint/version metadata
- missing telemetry or freshness checks

Then, based on that impact analysis, propose the hardening task.

Phase 2 — Upstream Hardening Task
I want a single implementation task that addresses:

A. Endpoint and scope source-of-truth hardening
- move toward centralized metadata-backed endpoint/scope resolution
- reduce or eliminate scattered hardcoded endpoint strings
- reduce or eliminate duplicated auth/token logic
- ensure one authoritative path for endpoint construction and scope lookup

B. Probe truthfulness
- separate:
  - configured
  - reachable
  - licensed/provisioned
  - confidence
- avoid optimistic “available” when the evidence is weak or inferred
- make probes explicit about official vs inferred vs deprecated endpoint confidence
- make weak/ambiguous probe results visible, not silently flattened into success

C. UI truthfulness
- make sure UI does not present optimistic success/availability states
- statuses shown to users should distinguish:
  - available
  - limited
  - unavailable
  - unknown
  - inferred
  - partially successful
- deployment/provisioning results must be truthful and granular

D. Drift detection and resilience
- ensure changes in UiPath endpoints/scopes/services are detectable
- include metadata freshness / verification concepts where appropriate
- improve logging/telemetry so drift or partial breakage is visible quickly

E. Flow safety
- this must NOT break the existing app flow
- preserve current successful behavior where possible
- treat current working behavior as the compatibility contract unless an intentional truthfulness correction is required
