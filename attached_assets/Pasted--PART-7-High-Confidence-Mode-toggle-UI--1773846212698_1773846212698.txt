PART 7 — High Confidence Mode toggle UI
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Add the High Confidence Mode toggle to the top of 
the chat section, pinned above the conversation 
history and visible at all times — not buried in 
settings.

VISUAL DESIGN

Match the existing dark theme exactly. The toggle 
sits in a compact bar directly above the chat input 
area or at the very top of the chat panel, depending 
on which feels less intrusive in the current layout. 
It must not push the conversation content down 
significantly — keep it slim, maximum 40px height.

The bar contains three elements inline:

Left: A small icon — a shield or checkmark symbol 
that visually communicates quality/validation. Use 
an SVG icon consistent with the existing icon style 
in the UI.

Centre: Label and mode selector
"Validation" followed by a compact three-option 
pill selector:
  [ Auto ] [ Always ] [ Off ]

The active option is highlighted in the brand orange 
colour used elsewhere in the UI (matching the 
Package and Deploy button colour visible in the 
screenshot). Inactive options are muted/grey.

Right: A contextual status chip that updates 
dynamically:
  When mode is Off:        no chip shown
  When mode is Auto:       
    Before generation:     grey chip "Ready"
    During scoring:        amber chip "Assessing..."
    Score below threshold: grey chip "Not needed"
    Score above threshold: amber chip "Will validate"
  When mode is Always:     orange chip "Active"
  During meta-validation:  pulsing orange chip 
                           "Validating..."
  After corrections:       green chip "✓ {N} fixes"
  After no corrections:    green chip "✓ Clean"

The status chip gives the user real-time feedback 
without requiring them to look at the streaming log.

BEHAVIOUR

The toggle persists the user's selection immediately 
on click — no save button needed. Store in user 
preferences alongside the existing settings.

When mode changes from Off to Auto or Always while 
a generation is already in progress, show a toast: 
"Will apply to next generation."

When mode is Auto and the confidence scorer runs, 
briefly animate the chip from "Assessing..." to 
either "Will validate" or "Not needed" with a 
subtle fade transition.

When meta-validation is running, the chip pulses 
gently to indicate activity — a slow opacity 
animation, nothing jarring.

PART 8 — Progress tracker messages
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The streaming progress log must never leave the 
user feeling stuck or uncertain. Every stage must 
emit a started message immediately when it begins 
and a completed message with context when it ends. 
There must never be a gap longer than 5 seconds 
without a new message appearing.

COMPLETE MESSAGE SEQUENCE

Emit these SSE events in order during generation. 
Each event has a stage, status, and summary that 
appears as a line in the streaming log panel.

Stage: sdd_validation
started:   "Reading your process specification..."
completed: "Process understood — {N} workflow steps 
            identified across {M} sub-processes"

Stage: decomposition
started:   "Mapping out the automation structure..."
completed: "Automation blueprint ready — {N} 
            workflows, {M} activities planned"

Stage: confidence_assessment (Auto mode only)
started:   "Assessing process complexity..."
completed: "Complexity score: {score} — {summary 
            from ConfidenceAssessment.summary}"

Stage: variable_resolution
started:   "Resolving variables and data types..."
completed: "{N} variables declared — types inferred 
            from naming conventions and spec"

Stage: template_resolution
started:   "Selecting activity templates from 
            catalog..."
completed: "{N} activities matched to catalog 
            templates — {M} packages required"

Stage: xaml_assembly
started:   "Assembling workflow files..."
  Emit a heartbeat every 3 seconds while this runs:
  heartbeat: "Building {workflowName}... ({N} of 
              {M} workflows complete)"
completed: "{M} XAML files assembled — 
            {totalActivities} activities across 
            {M} workflows"

Stage: catalog_validation
started:   "Validating activities against UiPath 
            catalog..."
completed: "Catalog validation complete — 
            compliance score {score} — {N} 
            auto-corrections applied"

Stage: quality_gate
started:   "Running quality checks..."
completed: "Quality gate passed — {N} warnings, 
            0 blocking errors"
  Or if warnings:
  warning:  "Quality gate complete — {N} warnings 
             flagged, package will open with 
             review notes"

Stage: meta_validation (when engaged)
started:   "Running extended validation — checking 
            {N} error categories..."
  Emit a heartbeat every 4 seconds:
  heartbeat: "Reviewing {workflowName} for 
              structural correctness..."
completed: "Extended validation complete — {N} 
            corrections applied, package verified"
  Or if clean:
completed: "Extended validation complete — no 
            issues found, package is clean"

Stage: packaging
started:   "Building UiPath package..."
completed: "Package ready — {fileSize}KB, 
            {N} workflows, {M} dependencies"

Stage: dependency_resolution
started:   "Confirming package dependencies..."
completed: "Dependencies confirmed — {packages} 
            at verified versions"

Stage: dhg_generation
started:   "Generating Developer Handoff Guide..."
completed: "Handoff Guide complete — {N} sections, 
            go-live checklist included"

Stage: complete
completed: "Your automation package is ready. 
            Download it or deploy directly to 
            UiPath Orchestrator."

HEARTBEAT MECHANISM

For any stage expected to take more than 5 seconds 
(xaml_assembly, meta_validation), implement a 
heartbeat emitter that fires every 3-4 seconds 
while the stage is running. The heartbeat message 
must be specific and meaningful — not a generic 
"Still working..." but actual progress context like 
which workflow is being assembled or which category 
is being validated.

Implement as a setInterval that clears when the 
stage emits its completed event:

function emitHeartbeat(stage: string, 
  getMessage: () => string, intervalMs: number) {
  const interval = setInterval(() => {
    emitPipelineEvent({
      stage,
      status: 'heartbeat',
      summary: getMessage(),
      timestamp: Date.now()
    })
  }, intervalMs)
  return () => clearInterval(interval)  // cleanup fn
}

The getMessage function is called fresh on each 
interval so it can reflect actual current progress 
(e.g. which workflow index is currently being 
assembled).

VISUAL TREATMENT IN THE LOG PANEL

Each message type has a distinct visual treatment:

started:   muted grey text, no icon, slight indent
heartbeat: muted grey text, animated ellipsis "..."
           appended, italic
completed: full brightness text, small ✓ prefix 
           in green
warning:   amber text, ⚠ prefix
failed:    red text, ✗ prefix

Completed messages show elapsed time for that stage 
right-aligned in the row:
  "Catalog validation complete — compliance 
   score 0.94"                          1.4s →

The timing gives the user a sense of what takes 
long and builds confidence that the system is 
working efficiently.

Active stage — the currently running stage — shows 
a subtle pulsing dot to the left of the message, 
matching the orange brand colour. This is the 
primary "not stuck" signal. At any given moment 
exactly one dot is pulsing.

On completion of the full pipeline, the log panel 
smoothly transitions to show the package ready 
state — the Package, Handoff Guide, and Deploy 
buttons become active with a brief fade-in. The 
log remains visible but scrolled slightly up, 
leaving the buttons visible without scrolling.
