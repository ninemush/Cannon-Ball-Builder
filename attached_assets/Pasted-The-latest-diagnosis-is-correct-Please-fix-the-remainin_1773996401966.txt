Observed root cause:
There is a deadlock between isGeneratingDoc and the deferred UiPath trigger effect.

Exact sequence:
1. The server sends docProgress: { started: true, docType: "UiPath" } during the chat stream.
2. The client treats this like document generation and calls startDocStreaming("UiPath"), which sets isGeneratingDoc = true.
3. The server then sends triggerUiPathGen: true and done: true.
4. The client preserves the UiPath progress message and does not clear isGeneratingDoc.
5. The deferred trigger effect sees hasPending=true but isGeneratingDoc=true, so it defers forever.
6. POST /api/ideas/:ideaId/generate-uipath never starts.

This is the key architectural issue:
For UIPATH_GEN, the chat stream is only an acknowledgement/trigger channel. The actual package generation runs in a separate POST request. Therefore the client must not treat UiPath package generation as an active "doc stream" blocker in the same way as PDD/SDD/DHG streaming.
