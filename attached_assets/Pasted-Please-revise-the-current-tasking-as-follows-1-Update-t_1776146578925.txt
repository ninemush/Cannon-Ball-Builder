### Root causes to address in #514

#### A. `compliancePass()` side effect creates duplicate `xamlEntries`

Problem:

* `Process.xaml` is generated once, then later rebuilt during spec-aware wiring.
* Some call sites invoke `compliancePass(...)` inline inside `syncXamlWrite(...)`.
* Because JavaScript evaluates arguments before the outer call, `compliancePass(...)` mutates tracking state first and pushes a new entry into `xamlEntries`.
* `syncXamlWrite(...)` then updates an existing entry but leaves the newly pushed duplicate orphaned.
* This creates duplicate tracked entries for `Process.xaml`.

Required fix:

* Audit **all inline `compliancePass(...)` calls passed directly into `syncXamlWrite(...)`**.
* In those cases, suppress tracking side effects by using `skipTracking: true`.
* `syncXamlWrite(...)` must remain the single owner of tracking in those code paths.
* Do this based on actual call sites in the current branch, not a hard-coded expected count.

#### B. `duplicate-file` remediation is treating bookkeeping duplication like XML corruption

Problem:

* The gap analyzer emits `duplicate-file` when duplicate tracked XAML entries exist.
* Current remediation treats that as a severe validation failure and replaces the file with a stub.
* For `Process.xaml`, this destroys the invocation chain and makes spec workflows unreachable.
* A duplicate bookkeeping record is not the same as malformed XML.

Required fix:

* Handle `duplicate-file` separately from real XML/content corruption categories.
* For duplicates, deduplicate entries rather than stubbing the file.
* Keep the latest valid content and remove extras.
* Stub remediation must remain reserved for true corruption categories such as `xml-wellformedness`, `malformed-quote`, and similar content-level failures.

#### C. `GetCredential` object-as-string crash in workflow assembly

Problem:

* `Username` and `Password` can arrive as `ValueIntent`-style objects instead of raw strings.
* Current logic later assumes string semantics and calls `.replace(...)`.
* This crashes assembly of workflows such as `CoupaLogin`, causing fallback to structural stubs.

Required fix:

* In both `crossCheckGetCredentialVariableTypes()` and the workflow tree assemblerâ€™s `GetCredential` handling, resolve object-shaped values to scalar strings before further processing.
* Reuse the same scalar-string resolution helper already used by deterministic generators.
* Guard string operations with type checks or explicit coercion after resolution.
* Never call `.replace(...)` on unresolved objects.

#### D. Safety net for critical infrastructure files

Problem:

* If `Main.xaml` or `Process.xaml` is stubbed, the invocation chain can be completely severed.

Required fix:

* Before stubbing critical orchestrator files such as `Main.xaml` or `Process.xaml`, extract and preserve existing `InvokeWorkflowFile` references from the original content where possible.
* Preserve those invoke relationships in the generated stub as a defense-in-depth safeguard.
* This is not the primary fix. It is a protective fallback.
