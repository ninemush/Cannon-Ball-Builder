Fix Process.xaml TextExpression block omission and Excel ExistingWorkbook requirement mismatch

What & Why
The latest CoupaInvoiceReconV2 package failed structurally for two primary reasons:

1. Process.xaml generation path omission
The rich generation path in server/xaml-generator.ts emits VisualBasic.Settings but omits:
- TextExpression.NamespacesForImplementation
- TextExpression.ReferencesForImplementation

The compliance/injection pipeline then assumes settings already exist and skips inserting the full TextExpression block, leaving Process.xaml without the namespaces/references containers needed for later import/reference injection. This directly causes:
- Process.xaml: failed to inject assembly references
- Process.xaml: failed to inject namespace imports

2. Excel Application Scope required-property mismatch
The imported catalog currently marks ExistingWorkbook as required for Excel Application Scope, but the generation/runtime stack does not model or emit it, and it does not appear to be universally required for normal usage. This creates repeated high-severity required-property failures that the generator cannot satisfy.

This task fixes both issues together because they are the current narrow blockers in the latest run.

Done looks like
- Process.xaml emitted by generateRichXamlFromNodes includes the full TextExpression namespace/reference structure needed by downstream injection.
- Process.xaml no longer fails with:
  - failed to inject assembly references
  - failed to inject namespace imports
- The rich generation path is structurally aligned with the buildXaml / working template paths for text-expression support.
- Excel Application Scope no longer fails validation solely because ExistingWorkbook is marked required when the current supported generation path does not use it.
- Latest Coupa benchmark run materially reduces structural invalidity caused by these two issues.

Out of scope
- Broad generator redesign
- Namespace map redesign
- Prompt/enricher changes
- REFramework invoke reconciliation
- General required-property fallback policy changes beyond the specific Excel mismatch here
- Broad catalog approval policy changes

Design rules
Template parity where required
If rich XAML generation needs downstream namespace/reference injection, it must emit the same structural containers that the working template paths emit.

No fake requiredness
A property must not remain marked required in the effective runtime metadata if the supported generation paths do not emit it and normal valid usage does not require it.

Minimal-delta fix
Fix the narrow structural omission and the narrow Excel metadata mismatch without expanding into broader metadata policy work.
