Critical Operation Prompt Tightening and Pre-Lowering Spec Normalization

What & Why
The downstream lowering layer already contains substantial machinery for critical operation handling, especially for mail:
- representation detection
- cluster tracking
- family locking
- provenance-aware collapse
- diagnostics
- package violation bridging

The real remaining gap is upstream:
1. the LLM is still allowed to emit more than one representation of the same logical critical operation
2. duplicate representations are reaching the lowering layer before any spec-level normalization removes them

This task should not rebuild family-selection or lowering abstractions upstream. It should reduce ambiguity before lowering by:
- tightening prompt instructions for critical operations
- adding a spec-level normalization pass that collapses duplicate representations of the same logical operation before they reach critical-activity-lowering.ts
- proving via diagnostics that lowering only receives the normalized representation

Goals
1. Reduce multi-representation critical operations at the prompt/spec level.
2. Normalize duplicate representations before lowering.
3. Feed cleaner, lower-entropy WorkflowSpec / WorkflowNode input into the existing lowering layer.
4. Reuse existing downstream family-locking infrastructure rather than rebuilding it upstream.

Done looks like
- Prompt instructions explicitly prevent the LLM from emitting both a narrative container and a concrete executable representation for the same logical critical operation.
- A spec-level normalization pass detects duplicate representations of the same logical operation before lowering.
- The normalization pass operates on existing WorkflowSpec / WorkflowNode structures, not new competing abstractions.
- Duplicate representations are collapsed deterministically where possible, or rejected before lowering if conflict cannot be resolved safely.
- Normalization diagnostics are emitted and wired into the existing diagnostics pipeline.
- Existing downstream lowering receives only the normalized canonical representation for the same logical operation.
- A trace artifact proves what lowering received after normalization.

Out of scope
- Rebuilding family-selection infrastructure upstream
- Adding new parallel mail family abstractions
- Broad catalog/filter alignment
- XAML namespace/prefix fixes
- Post-emission repair changes
- Full decomposition redesign

Tasks

1. Tighten prompt instructions for critical operations
Update the relevant spec-generation prompt so the model is explicitly told:
- do not emit both a narrative container and a concrete executable step for the same logical operation
- if Try/Catch is needed, represent it in a structured machine-lowerable form, not as prose text plus a duplicate concrete step
- for critical operations like mail send, task create/wait, invoke workflow, and Data Service CRUD, emit one canonical representation only
- preserve required business fields like To, Subject, Body, connection/context fields in that one representation

2. Add a pre-lowering spec normalization pass
Add a normalization step between decomposition/spec generation and critical-activity-lowering that:
- inspects existing WorkflowSpec / WorkflowNode structures
- detects duplicate representations of the same logical critical operation
- collapses them into one canonical representation when deterministic
- rejects them when there are conflicting non-empty values that cannot be resolved by an explicit precedence rule

This pass must not invent new operation-family models. It should prepare cleaner input for the existing lowering code.

3. Eliminate narrative pseudo-container duplication
If the spec contains both:
- a narrative Try/Catch-style representation
- and a concrete executable step
for the same logical operation,
the normalization pass must:
- preserve one canonical representation
- remove or normalize the duplicate
- or reject the operation if deterministic collapse is not possible

4. Preserve provenance and conflict handling
When collapsing duplicate representations, record:
- which source representation supplied each preserved field
- which duplicate representations were removed
- which fields conflicted
If two representations provide conflicting non-empty values for the same required property and no deterministic precedence rule resolves them, reject the operation instead of guessing.

5. Emit normalization diagnostics
Add:
criticalOperationSpecNormalizationDiagnostics

with per normalized/rejected operation:
- workflow/file
- operationClusterId
- operationType
- representationsDetected
- canonicalRepresentationChosen
- preservedFields
- droppedRepresentations
- conflictFields
- deterministicResolutionApplied
- rejected: true/false
- remediationHint

Also include summary counts:
- totalCriticalOperationClusters
- totalDuplicateRepresentationsDetected
- totalNormalizedSuccessfully
- totalRejectedForConflict
- totalNarrativePseudoContainersRemoved

6. Prove active-path adoption
Emit a second trace artifact:
criticalOperationNormalizationTrace

with per workflow:
- workflow/file
- preNormalizationRepresentationCount
- postNormalizationRepresentationCount
- normalizedOperationClusterIds
- loweringInputOperationClusterIds
- loweringSawOnlyNormalizedRepresentation: true/false

The executor must prove that critical-activity-lowering.ts consumed the normalized output, not the original pre-normalized spec.

7. Integrate with existing lowering
Ensure the normalized WorkflowSpec / WorkflowNode output is what critical-activity-lowering.ts consumes.
Do not add a competing abstraction layer.

8. Add focused regression tests
Cover only the new upstream behavior:
- mail operation emitted as both narrative Try/Catch and concrete send step normalizes to one representation
- To/Subject/Body are preserved after normalization
- conflicting non-empty values cause rejection, not guessing
- downstream lowering input contains only the normalized representation
- no new parallel family abstraction is introduced
- normalization trace proves lowering saw the normalized spec
