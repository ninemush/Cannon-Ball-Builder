FIX 1 — Move ampersand sanitizer outside 
the quality gate failure block
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

In uipath-integration.ts, the ampersand 
sanitizer at lines 1486-1491 is inside 
the if (!qualityGateResult.passed) block 
at line 1405. This means it only runs 
when the quality gate fails — if the 
quality gate passes with warnings, the 
sanitizer is skipped entirely.

The sanitizer must run unconditionally 
on every generation regardless of quality 
gate result.

Move the ampersand sanitizer loop to 
run BEFORE the quality gate check, not 
inside its failure branch. The correct 
order is:

  1. Generate XAML
  2. Run ampersand sanitizer on all 
     xamlEntries (unconditionally)
  3. Run quality gate on sanitized XAML
  4. If quality gate fails, run remaining 
     auto-remediation
  5. Run final validation

This is a structural fix — move the 
sanitizer block, do not duplicate it. 
After the move, the ampersand sanitizer 
should NOT appear inside the quality 
gate failure block anymore.

Also fix the related VB expression 
escaping issue. The failing content is:

<InArgument x:TypeArguments="x:String">
  "https://...?lat=" & str_Lat & 
  "&lon=" & str_Lon
</InArgument>

Both the VB concatenation & operators 
and the URL query parameter & characters 
need to be &amp; in XML text content. 
The XML parser unescapes them at parse 
time so VB/UiPath sees bare & at runtime.

The existing sanitizer regex already 
handles this correctly — it just needs 
to run before final validation. Moving 
it out of the quality gate block is 
the complete fix.

GAP A — sendPipelineEvent missing flush
In document-routes.ts, the sendPipelineEvent 
function at line 1061 does not call flush 
after writing. Add flush:

  const sendPipelineEvent = (
    event: PipelineProgressEvent
  ) => {
    res.write(
      `data: ${JSON.stringify(
        { pipelineEvent: event }
      )}\n\n`
    )
    if (typeof (res as any).flush === 
        'function') {
      ;(res as any).flush()
    }
  }

GAP B — Node.js fetch() buffering SSE 
from internal generate-uipath call

In chat/routes.ts around line 1232, the 
chat handler reads the internal 
generate-uipath SSE stream via 
fetch().body.getReader(). Node.js fetch 
may buffer chunks from this stream, 
causing pipeline events to arrive late 
or not at all during the generation.

Fix: Add a Transfer-Encoding: chunked 
header to the generate-uipath endpoint 
response in document-routes.ts to hint 
to Node.js that the response should not 
be buffered:

  res.setHeader(
    'Transfer-Encoding', 'chunked'
  )
  res.setHeader('X-Accel-Buffering', 'no')

Add these headers immediately after the 
existing SSE headers are set in the 
generate-uipath route (after the 
Content-Type and Cache-Control headers).

Also in the chat/routes.ts internal fetch 
call around line 1195, add these request 
headers to signal streaming intent:

  headers: {
    'Accept': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'X-Accel-Buffering': 'no'
  }
