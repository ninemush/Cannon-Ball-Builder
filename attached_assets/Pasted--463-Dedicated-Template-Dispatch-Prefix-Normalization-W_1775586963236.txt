#463 - Dedicated Template Dispatch Prefix Normalization

What & Why
Recent benchmark evidence now shows repeated required-property failures on activities that already have dedicated template support, especially:
- Log Message missing required property "Message"
- Send SMTP Mail Message missing required property "Body"

A prior evidence-based isolation on the Birthday benchmark already confirmed that prefixed template names such as `ui:SendSmtpMailMessage` can bypass dedicated template dispatch in `resolveActivityTemplate()`, causing the activity to fall through to a generic/dynamic path that omits schema-critical properties.

This task must implement a narrow dispatch-boundary fix:
normalize namespace-prefixed template names before dedicated template dispatch so activities with dedicated templates are routed through the correct template consistently.

This is not a broad assembly-time schema redesign. It is a targeted fix for a likely shared failing path.

Done looks like
- Activities with dedicated templates are dispatched correctly even when `node.template` arrives with a namespace prefix.
- `SendSmtpMailMessage` no longer bypasses its dedicated template due to prefixed template names.
- `LogMessage` no longer bypasses its dedicated template due to prefixed template names.
- Benchmark runs show material reduction in:
  - missing `Body` on SMTP mail activities
  - missing `Message` on LogMessage activities
- The fix is bounded to template dispatch and does not broaden into family-wide schema rewrites.

Out of scope
- Broad required-property schema authority redesign
- Graph connectivity fixes
- Workflow contract/interface work
- Typed-property-object boundary work
- Prompt engineering
- Rewriting individual dedicated templates unless the investigation proves they are also broken
- Quality gate or final validation rule changes

Design rules
1. Dispatch-boundary normalization only
Normalize namespace prefixes at the point where template dispatch decisions are made.
Do not scatter prefix stripping across unrelated downstream layers.

2. Preserve explicit template semantics
The fix must preserve the original intended activity identity. It should normalize only the dispatch key, not mutate unrelated business data.

3. Dedicated-template first
If a dedicated template exists for the normalized activity name, it must win over generic/dynamic emission paths.

4. No broad assumptions
Do not assume every missing-property defect is caused by prefix mismatch. This task is only for the dispatch-boundary path shared by the benchmark-hit activities.

Tasks
1. Confirm the active failing path on the current benchmark
Using the latest benchmark artifacts available for this use case, confirm whether the repeated:
- LogMessage missing Message
- SendSmtpMailMessage missing Body
failures are entering `resolveActivityTemplate()` with prefixed `node.template` values.

If existing artifacts are insufficient, reproduce with a focused regression test or trace.

2. Normalize template name before dedicated dispatch
In `resolveActivityTemplate()`, normalize `node.template` into a dispatch key by stripping any leading namespace prefix (for example `ui:` or similar) before the dedicated template dispatch chain runs.

This normalization should affect dispatch matching only.
Do not break any places that still need the original template string for diagnostics or metadata.

3. Ensure dedicated template dispatch wins after normalization
Verify that after normalization:
- `ui:SendSmtpMailMessage` dispatches to `resolveSendSmtpMailMessageTemplate()`
- `ui:LogMessage` dispatches to the dedicated LogMessage path
- other existing dedicated templates with prefixed names also route correctly when a dedicated template exists

4. Keep generic/dynamic fallback unchanged for truly unsupported activities
If no dedicated template exists after normalization, preserve the current fallback behavior.
Do not turn this into a broad activity-family redesign.

5. Add focused diagnostics
Extend an existing lightweight reporting surface so benchmark/debug runs can show:
- original template name
- normalized dispatch key
- whether dedicated template dispatch was used
Keep this lightweight. No new diagnostics framework.

6. Add focused regression tests
Add tests covering:
- bare `SendSmtpMailMessage` still dispatches correctly
- prefixed `ui:SendSmtpMailMessage` now dispatches to the same dedicated template
- bare `LogMessage` still dispatches correctly
- prefixed `ui:LogMessage` now dispatches to the same dedicated template
- unsupported prefixed activity still falls back normally
- benchmark-style missing Body / missing Message path no longer reproduces when the only cause was dispatch prefix mismatch
